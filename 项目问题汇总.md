## Chat

一款集成 AI 机器人的即时通讯应用。该项目后端基于 Go 实现用户、好友、群聊、黑名单、消息会话等核心能力，使用 WebSocket 搭建高实时双向通信链路，引入 Kafka 作为消息缓冲、对消息收发进行异步化、为缓冲失败重试提供支持；同时为 AI 机器人接入 MCP 工具调用、RAG 检索增强与 SSE 流式响应，实现智能问答与工具执行能力。

请简述 WebSocket 网关的设计思路。如何处理海量长连接的读写并发？

- 架构模式 : 采用 Hub 模式 管理连接，维护一个并发安全（使用mutex）的 Clients 映射表（ map[string]*Client ）。
- 读写分离 : 每个客户端连接启动两个 Goroutine，一个负责读，一个负责写。
- 心跳保活 : 实现应用层 Ping/Pong 机制，定期清理僵尸连接，释放服务器资源。读侧设置超时并在收到 Pong 时续期，写侧定时发送 Ping 并设置写超时，连接断开时发送 Close 控制帧。
- 用户登陆时，将请求升级为ws，并将userID存入map映射中，开启两个goroutine分别进行读写，读到消息直接写入 Kafka（ ChatWriter ），由 Kafka 消费端负责写入数据库、分发（判断消息类型），消费者会将消息发送给对应client的Sendback通道。写goroutine监听这个通道，有消息则发回。

你的“消息存储与缓存策略”具体是如何实现的？为什么选择 Cache-Aside 模式？

- 写入流程（Write Path）- 更像是 write through
  
  - 用户消息先落 MySQL。
  
  - 同步写入 Redis ZSet 缓存：
    
    - 单聊 key： message_zset_{minUid}_{maxUid} （做了 ID 排序，避免 A->B / B->A 双 key）
    - 群聊 key： group_message_zset_{groupId}
    - score = created_at.UnixNano() ，保证严格时间有序； member = 消息 JSON 
  
  - 通过 Pipeline 原子执行 ZADD + ZREMRANGEBYRANK（插入一个元素后，删除 0 到 倒数第501元素） + EXPIRE ：
    
    - 保留最近 500 条热点消息，控制缓存体积。
    - 设置 TTL（当前沿用项目超时配置）。
  
  - 写缓存失败那么 发送消息到消息队列进行重试

- 读取流程（Read Path）
  
  - 先查 Redis ZSet（ ZRANGE 0 -1（返回所有数据） ），命中则直接返回（毫秒级）。
  
  - 未命中或空缓存时，回源 MySQL 按时间排序查询。
  
  - 将 DB 结果批量回填到 ZSet（ ZAddMessagesWithTTL ），完成一次完整 Cache-Aside 闭环。
  
  - 这样下一次同会话读取直接命中缓存。

- 为什么选择 Cache-Aside
  
  - 聊天历史属于典型“读多写多但热点集中”的场景，Cache-Aside 能让 DB 做最终真相，Redis 负责热点加速。
  
  - 业务侧可控性高：回源、回填、失效策略都可按会话维度精细控制。
    
    - 与 ZSet 组合后天然支持按时间排序、滑动窗口、后续游标分页，比“整段 JSON 数组 Get/Set”更稳定、更省 CPU。

WebRTC 是 P2P 的，为什么还需要后端参与？你的信令流程是怎样的？

媒体仍然是 P2P，也就是建立连接之后，无需服务进行中断。但 信令交换、状态仲裁 必须依赖后端。

1) 呼叫发起（A → 后端 → B）
- A 点击呼叫后，先向后端发送 CALL_INVITE(callId, from=A, to=B) 。

- 网关节点把该事件写入 Kafka 的 webrtc_signal_topic （key 使用 callId 或 sessionId ，保证同一通话信令有序）。

- 消费端（信令服务）消费到 CALL_INVITE ：
  
  - 校验权限（是否好友/是否允许通话）。
  - 查询/更新通话状态机： IDLE -> RINGING 。
  - 若 B 忙线或已有进行中通话，直接返回 BUSY 给 A，并终止该 callId。
  - 否则把 CALL_INVITE 定向推送给 B（通过 WebSocket 下发）。
2) 接听/拒绝（B → 后端 → A）
- B 收到 invite：
  
  - 拒绝：发送 CALL_REJECT(callId) ；服务端状态机 RINGING -> ENDED ，通知 A。
  - 接听：发送 CALL_ACCEPT(callId) ；服务端状态机 RINGING -> CONNECTING ，通知 A 进入协商阶段。
3) SDP 协商（Offer/Answer）
- A 生成 offer：发送 SDP_OFFER(callId, sdp) → Kafka → 信令服务 → 转发给 B。

- B 收到 offer 后 setRemoteDescription ，生成 answer：发送 SDP_ANSWER(callId, sdp) → Kafka → 转发给 A。

- 当双方 SDP 交换完成，状态机进入 CONNECTING （或直接由服务端在满足条件后标记为 IN_CALL ）。
4) ICE Candidate 交换（Trickle ICE）
- A/B 持续上报 ICE_CANDIDATE(callId, candidate) → Kafka → 对端转发。

- 客户端侧按需做 candidate 缓冲（remoteDescription 未就绪则暂存），就绪后再 addIceCandidate 。
5) 通话中/挂断/异常（状态闭环）
- 任意一方挂断：发送 CALL_HANGUP(callId) → 状态机 IN_CALL -> ENDED ，通知对端清理资源。

- 超时：例如 RINGING 超过 30 秒无人接听，服务端自动 TIMEOUT 并置 ENDED ，通知双方。

- Busy：服务端在 CALL_INVITE 或 CALL_ACCEPT 处做仲裁，保证同一用户同一时间只处于一个 active call。

项目中提到的 RAG（检索增强生成）是如何实现的？文档切片策略是什么？

- 入库流程 : 用户上传文档 -> 文本切片 （按段落/长度切分，支持重叠窗口） -> 调用 Embedding 模型向量化 -> 存入 Redis Stack。

- 检索流程 : 用户提问 -> 问题向量化 -> 在 Redis 中进行向量相似度搜索（Vector Search） -> 召回 Top-K 相关片段。

- 生成流程 : 将召回片段作为 Context 拼接到 Prompt 中，输入 LLM 生成回答。

- 切片优化 : 实现了带重叠（Overlap）的切片策略，避免语义被切断，提高召回准确率。

kafka在本项目中的作用？

1) 作为 WebSocket 上行消息的缓冲队列（削峰填谷）

        WebSocket 收到前端消息后， Client.Read() 在 kafka 模式直接把消息写入 chatTopic         ,而不是同步在读协程里做落库/分发，避免 WebSocket 读被 DB/业务拖慢。        

2. 作为聊天消息处理的异步工作流（消费端统一处理：落库 + 推送 + 缓存）
   
   KafkaChatServer.Start() 消费 chatTopic ，在消费端完成：
   
   - MySQL 持久化
   - WebSocket 推送给在线用户（ SendBack -> Client.Write() ）
   - Redis 缓存更新（你现在已改为 ZSet 缓存）

3. AI 请求与普通消息解耦（专属 AI Topic）
   
   - 普通消息流走 chatTopic ，AI 请求走 aiTopic ：
     - 写入 AI Topic：当消息接收方是 AI Bot 时，把请求写入 AITopic 。 
     - AI 消费者： AIService.StartConsumer() 消费 AIReader ，异步调用 LLM/RAG/MCP 等逻辑处理。

4. 为了满足webrtc消息的实时性，单独建立rtcTopic

5. redis缓存重试

怎么使用eino框架的？

我把 AI 助手的关键步骤抽象成 Eino 的 Component 节点 ，再用 compose.Graph 把数据流串起来，形成“可编排、可复用、可插拔”的工作流： START -> preprocess -> (route) -> rag_retrieve/mcp_call -> prompt_build -> chat_model -> postprocess -> END 。

- 节点类型对应关系：
  - ChatModel 节点 ：用 Eino 的 ToolCallingChatModel 抽象，底层可切 OpenAI/Ollama 的实现（你项目已有 openai.NewChatModel / ollama.NewChatModel ）。 model.go
  - RAG 节点 ：Retriever/Embedding/Indexer 作为独立节点（检索节点只负责“召回文档”，不掺业务逻辑）。 使用eino完成检索，只有在判断需要用rag的时候才调用。
  - 业务节点 ：比如“解析输入/拼 prompt/解析工具调用 JSON/落库”等，用 Lambda 节点包一层函数实现。

Rag怎么实现的？

RAG 分两条链路： 入库（Indexing） 和 查询（Retrieval + Generation） 。

1) 入库链路（Indexing）：文档 → Chunk → 向量 → Redis
- 文档切分（Chunking） ：上传文件后按段落/长度切块并做重叠窗口，避免语义被切断（你项目现在是 max 1000 chars，overlap 150）。

- Embedding 节点 ：用 eino-ext/components/embedding/openai 创建 Embedder，把每个 chunk 转成向量。 rag.go

- Indexer 节点（Redis） ：用 eino-ext/components/indexer/redis 把 chunk 的文本与向量写进 Redis（Hash + vector 字段），并通过 DocumentToHashes 定义 Redis 存储结构。 rag.go

- 在 Graph 的视角： parse_file -> chunk -> embed -> index_store （可串成一个 indexing graph，或作为后台任务执行）。
  
  查询链路（Query）：问题 → 向量检索 → 上下文增强 → LLM 生成

- 问题向量化 ：复用同一个 Embedder，把用户 query 转成向量。

- Retriever 节点（Redis 向量召回） ：用 eino-ext/components/retriever/redis 对 Redis 向量索引做 TopK 相似度检索，拿到最相关的 chunk 文档列表（内容/metadata/距离）。 rag.go

- Prompt 构建节点 ：把召回的 TopK 文本片段拼成 context，和用户问题一起组成最终 prompt（常见是“你只能基于以下资料回答…”）。

- ChatModel 节点生成 ：把 messages 交给 ChatModel.Generate/Stream 输出最终答案（可流式）。

- 在 Graph 的视角： preprocess -> embed(query) -> retrieve(topK) -> prompt_build -> chat_model -> postprocess 。

分块策略

- 入口在 rag.go:IndexFile ，核心调用 splitTextChunks(text, 1000, 150) ： rag.go
- 具体策略是“ 段落优先 + 长度兜底 + 重叠窗口 ”：
  - 归一化换行 ：把 \r\n/\r 统一为 \n ，避免跨平台差异影响切块。 rag.go
  - 按双换行分段 ： strings.Split(text, "\n\n") ，优先保持段落语义完整。 rag.go
  - 最大长度约束 ：目标 chunk 最大约 1000 rune 。如果某段落本身超过 1000，就按 rune 硬切成多个子段（ splitByRunes ）。 rag.go
  - 重叠（overlap） ：每个 chunk flush 后，会把上一个 chunk 的末尾 150 rune 作为下一个 chunk 的开头，减少“信息被切断导致召回不到”的问题。 rag.go
  - 文档元数据 ：每个 chunk 存成一个 schema.Document ，带 source 和 chunk 序号元数据。 rag.go

Rag优化

- TopK 分层 ：先取更大的 TopK （如 20 50）做“粗召回”，再用轻量 rerank（可用 LLM 小模型或规则打分）筛到最终 5 10 段；单纯 TopK=5 容易漏掉关键片段

- 动态 chunk 大小 ：当前固定 1000/150 对所有文档都一样；可以按文档类型（FAQ/代码/长文）调整：代码类适合更小 chunk + 更大 overlap，长文适合段落/标题结构切分。

- 基于结构切分 ：对 Markdown 可以按 # 标题层级切分（段落只是兜底），召回会更“主题化”。

1. 先建立SSE连接

2. 用户给 AI 提问（通过websocket）后，该请求会被写到 Kafka 的 AI Topic ，供 AI 服务异步消费

MCP (Model Context Protocol) 是什么？你的 Agent 是如何通过它调用外部工具的？

- MCP 是一套标准化的“模型 ↔ 工具/数据源”的协议，可以把外部工具（天气查询、数据库查询、搜索、企业内部系统等）以统一的接口暴露给大模型调用。
- 实现机制 :
1. Server 端 : 基于 mcp-go 封装工具（如“查询天气”），本地启动server，暴露标准接口。
2. Client 端 : Agent 识别用户意图（如“北京天气如何”），构造 MCP 请求。
3. 调用 : 通过 MCP Client 连接 Server，执行工具函数，获取结果 JSON。
4. 回传 : 将工具结果回传给 LLM，由 LLM 组织自然语言回复。

SSE流程

建立 SSE 长连接

- 前端发起 GET /aiSSE?userId=... （通过 query 指定订阅用户）。
- Controller 设置 SSE 必需的 Header：
  - Content-Type: text/event-stream
  - Connection: keep-alive
  - Cache-Control: no-cache
  - X-Accel-Buffering: no （避免 Nginx 缓冲导致“假流式”）
- 然后 AISSEHub.Subscribe(userId) 取到一个 chan string ，进入 c.Stream(...) 循环：每次从 channel 读到 chunk 就 c.SSEvent("message", msg) 推给前端。

用户向 AI 发送问题（走 WebSocket/Kafka）

1. 用户在聊天界面把消息发给 AI（接收方是 AI Bot），消息从 WebSocket 网关 进入系统（ client.Read() 写 Kafka 或 channel）。 client.go

2. 服务端识别这是 AI 消息后，会把请求写入 AI Topic （ aiTopic ）用于异步处理（避免阻塞聊天主链路）。 server.go
   
   B) AI 处理与 SSE 流式输出（走 HTTP SSE）

3. AIService.StartConsumer() 消费 aiTopic ，解析 ChatMessageRequest ，起 goroutine 处理该请求。 ai_service.go

4. 调用 helper.StreamResponse(...) 获取大模型的流式输出；每产生一个 chunk，就执行回调 AISSEHub.Publish(userId, chunk) ，SSE 连接立刻收到并渲染“打字机效果”。 ai_service.go

5. 结束时发布 "[DONE]" ，前端可据此结束 loading 状态。 ai_service.go
   
   C) AI 最终结果如何和“聊天消息体系”兼容（走 Kafka chatTopic + WebSocket）

6. AI 最终会把“完整回复”封装成一条普通聊天消息（ Type: Text ， SendId = AI_BOT_UUID ， ReceiveId = userId ），写回 chatTopic 。 ai_service.go

7. 然后由聊天消费者（ KafkaChatServer 或 channel server）走统一链路：
- 落库 MySQL
- 更新 Redis 历史消息缓存
- WebSocket 推送给前端（保证聊天列表/历史记录可回溯）

为什么 AI 回复要使用 SSE（Server-Sent Events）？它与 WebSocket 有什么区别？

- 天然匹配“打字机流式输出” ：AI 推理过程中会不断产生 token，服务端可以边生成边推送给前端展示。你项目里就是在 AI 消费端拿到 chunk 后 AISSEHub.Publish(userId, chunk) ，由 SSE 连接实时推给前端。 ai_service.go 、 ai_sse_controller.go

- 实现成本低 ：SSE 本质是一个保持不断开的 HTTP 响应流（ text/event-stream ），服务端只要不断写事件即可，浏览器端用 EventSource （或等价方案）就能接收。

- 断线重连更友好 ：SSE 天生支持重连语义（标准里有 Last-Event-ID），适合移动端/弱网下 AI 输出这种长时间连接。
  
  它与 WebSocket 的区别

- 通信方向
  
  - SSE： 单向 （服务端 → 客户端），客户端不能通过 SSE 回传数据。
  - WebSocket： 双向 （客户端 ↔ 服务端），适合聊天消息、WebRTC 信令这种需要双向交互的场景。

- 协议与通道
  
  - SSE：基于 HTTP（更容易穿透代理/负载均衡配置），是“长连接 HTTP 响应”。
  - WebSocket：独立协议升级（Upgrade），通常需要更明确的网关/代理支持。

引入 Kafka 后，系统的抗压能力体现在哪里？如果数据库抖动会发生什么？

- 削峰填谷 : Kafka 作为缓冲层，能暂存突发的海量消息。即使后端处理速度（如写库、AI 推理）暂时跟不上，消息也不会丢失，而是堆积在 Topic 中等待消费。
- 异步解耦 : 消息持久化逻辑从 WebSocket 的读写协程中剥离，移至 Kafka Consumer 端异步执行。
- 容错性 : 如果 MySQL 短暂抖动或宕机，Consumer 虽会报错重试，但不会阻塞前端用户的 WebSocket 发送操作，保证了聊天功能的 高可用性 。

在 Kafka 消费者处理逻辑中，你是如何保证消息不丢且不重？

- 保证不丢：
  - - 写端改成 RequiredAcks = kafka.RequireAll （或至少 RequireOne），并配置重试。
    - 读端改成“处理成功后手动提交 offset”（commit-after-processing）。
- 保证不重：
  - - 让客户端/服务端生成稳定的 message_id （请求里携带），DB 用唯一索引保证幂等写入（或 Redis SETNX 去重）。
    - 即使重复消费，也只会命中一次入库和一次推送。

如何做到横向扩容？

- 多个服务器部署代码，nginx用于选择服务器实例将请求发送。
* Kafka 作为消息总线 ：所有实例都消费 chatTopic ，收到消息后只对“本机在线连接”推送。

* 每个实例使用不同 GroupID ，保证每台机器都能看到全量消息，然后只给本机在线用户推送。

消息状态如何定义（未发送/已发送）？失败如何重试？

没有失败重试逻辑，避免消息顺序出问题。

同一会话消息如何保证有序？Kafka 的 key 该怎么设计？

同一会话的消息必须落到 Kafka 的同一分区 ，这样 Kafka 才能保证顺序。
 实现方式就是**分区键（Kafka Key）必须与会话强相关，sessionID**。

消息幂等怎么做（避免重复入库/重复推送）？

1. 前端发送消息时携带消息唯一标识

2. 数据库用唯一索引保证不重复

3. redis中保存唯一标识，用于消费消息进行一层判断

4. 如果redis中已过期，那么当插入数据库失败时，则不推送该消息。

Redis 缓存失效与脏读怎么处理？

缓存失效等下次请求重新读数据库更新

脏读则增加消息队列对写入redis失败进行重试。

Redis 缓存与 MySQL 如何保证最终一致？

- 先写 DB，再写缓存 ：落库成功后更新缓存，避免“缓存有但 DB 无”的不一致。
- 缓存不成功，进行重试
- 回源回填 ：读取时 miss 会从 DB 拉最新数据并回填 Redis，形成“自愈”。

WebSocket 断线重连怎么处理？

后端通过ws心跳判断断线，重连逻辑在前端 onclose 做指数退避重连

心跳机制怎么判断僵尸连接？断开后资源如何清理？

读超时或 Ping/Pong 失败就判定为僵尸连接 ，读写协程都会退出。

断开后在map中删除对应用户，关闭channel。

Kafka 消费者offset 如何管理

- 手动提交 ：消费成功（落库 + 推送成功）后再 CommitMessages ，避免“先提交后处理失败”导致丢消息。

- 幂等保障 ：即使发生重复消费，也能通过 messageId 去重，保证最终结果正确。

AI chunk 丢失/乱序如何处理？是否需要聚合与重传？

- 不做重传 ，不保证每个 chunk 必达。
- 最终一致靠完整消息回写 ：即使流式过程丢了几个 chunk，聊天列表最终会拿到完整 AI 回复。

前端保证sse消息和ws消息不会重复

- 把 SSE 输出渲染在同一个“临时气泡”里

- 收到最终 WebSocket 消息后 替换/固化 该气泡，而不是新增一条

向量索引参数（HNSW/距离度量）如何选？

1) **HNSW**
- 适合中等到大规模向量检索，查询速度快、召回效果可调。

- 对 RAG 场景更现实：你需要低延迟而不是 100% 精确检索。
2) **COSINE**
- OpenAI 等主流 embedding 默认更适合用余弦相似度（尤其是归一化向量）。

用户鉴权如何做？token 过期如何处理？

        用户鉴权怎么做

- 登录成功后生成 JWT（包含 userId/username/exp），返回给前端。

- 前端把 token 存储在本地，并在后续请求的 Authorization: Bearer <token> 中携带。

- 服务端全局中间件校验 token，并将用户信息写入上下文。
  
  token 过期如何处理

- 服务端 ParseToken 返回失败 → 直接 401，提示重新登录。

- 前端检测到 401 → 清理登录态 → 跳转登录页。

- 如果要无感续期，可以加 Refresh Token 机制（再加一个token，过期时间长，用于发给后端申请新的jwt）。

SSE 与 WebSocket 的鉴权如何避免“随便订阅别人 userId”？

核心原则：不信任 query 参数的 userId，而是用鉴权 token 解析出的身份。

SSE

- 客户端在 header/cookie 里带 JWT（或 session）

- 服务端验证 token，得到 userId

- 忽略 query 的 userId ，用 token 中的 userId 订阅
  WebSocket

- 连接时带 token（Header/Cookie/URL 参数）

- 服务器校验 token 后，再允许升级连接

- 连接绑定的 userId 只能来自 token，而不是 URL 参数

## Track

我做的是 AI生成任务 全链路事件追踪平台，目标是把一次AI生成任务从“客户端发起→服务端审核/前置处理→AIP生成→产物审核→交付”的每个关键事件统一存储，解决出现问题查询慢、需要跨系统查的问题。研发、测试人员可以在系统上输入任务id以获取对应任务的全事件流程，如果某事件失败会携带完整的错误信息。 也可以输入user_id得到该用户的最近任务。

架构上采用上报-处理-存储三层架构，完成客户端-服务端事件贯通、使用消息队列的跨地域复制实现全球节点数据汇聚，通过存储索引设计实现秒级查询，最终达到全链路可视化。

- 为什么采用“上报/处理/存储”三层架构？
  
  - - 解耦业务与追踪 ：上报层只负责把事件“可靠、低成本地送出去”，业务侧不需要关心写库细节；处理层统一做解析、聚合、扇出；存储层专注承载不同查询形态（TokaDB实时、Abase索引、Hive离线）。
    - 削峰填谷与抗抖动 ：AIGC链路事件高并发，写请求多，上报直接写库容易把数据库打爆；引入MQ作为缓冲，处理层可按消费能力水平扩展，平滑写入压力。
    - 全球化汇聚更简单 ：上报发生在全球各节点，处理层利用EventBus跨地域同步把数据汇到中心消费集群（SG），存储落在统一地域，最终形成全局视图。

- 为什么不统一用同一个id
  
  - 以最小的代码侵入实现日志上报功能。
  
  - 现有系统已有主键体系 ：服务端大量逻辑天然围绕 task_id构建；客户端之前就有 trace_id 。
  
  - 成本更小：保持客户端 trace_id 、服务端 task_id 不变，只在关键点写 trace_id ↔ task_id 索引，查询时再关联，能以**最小改动**达成“统一视图”。

- 都有哪些事件？
  
  - 客户端：用户上传、点击生成、发起批量请求上报 、用户取消、用户断网、结果成功渲染。客户端可以对不同trace_id的事件一起聚合发送。
  
  - 服务端：输入审核、进行输入prompt的强化扩写、素材解析、开始生成、生成失败/成功、输出审核、结果交付。

- 分成客户端批量上报与服务端SDK两种上报方式的主要原因是什么？
  
  - 客户端需要减少耗电而批量发送，服务端不需要，不必再加深复杂度。
  
  - 解耦客户端事件和服务端事件的处理，服务端不必再对客户端事件进行处理，减轻服务端负担，客户端通过发送请求到FaaS进行上报，两者分别提交自己的事件即可。

- trace_id 和 task_id怎么联通？
  
  - 客户端把多个事件批量发到 /trace 接口，里面都有 trace_id 。上报到消息队列中
  
  - 客户端调用 /generate 时把 trace_id 放到 extra 里一起传给服务端。服务端接收后把 trace_id ↔ task_id 映射写入redis
  
  - 消费端取出消息后，进行处理将trace_id转为task_id再进行存储

- 没有这个系统前，为什么会有排查困难的问题？
  
  - - 链路太长 ：一次AIGC任务要经过配额、输入审核、前置处理、模型生成、输出审核、交付，任一环节都可能失败。
    - 数据分散在多系统 ：**客户端日志、服务端日志、审核服务日志**、**模型平台日志、不同地域的数据都在不同地方**，查一次问题要“到处翻”。
    - 缺少统一主键贯通 ：没有 trace_id ↔ task_id 映射时，客户端和服务端任务难以快速对齐
    - 跨地域消息存在时间格式问题，查看费力
  - 现在支持在一个系统通过输入task_id直接查看所有事件，并且可以通过user_id查看用户最近任务、事件。

- 消息队列的Topic有哪些？
  
  - 分为客户端事件和服务端事件两个Topic
  
  - 主要是因为两者的处理流程不同，对于客户端事件需要进行trace_id 到 task_id的映射过程。
  
  - Topic 数 ：2（client、server）
  
  - 分区数建议 ：
    
    - client topic： 24 分区
    
    - server topic： 48 分区 （服务端事件通常更大）
  
  - 消费者并发建议（同一 consumer group） ：
    
    - client consumer 并发： 16~24
    
    - server consumer 并发： 36~48
    
    - 消费者会进行动态扩缩容，配置最大最小值。
    
    - 固定消费者数量然后进行测试

- 消费端怎么处理消息？
  
  - - 1）拉取消息 ：从 MQ 按分区并发拉取客户端/服务端事件。
    - 2）解析与校验 ：反序列化、字段校验（tenant/business/scene/source）timestamp是否有效。
    - 3）关联补全 ：若只有 trace_id ，先查映射拿 task_id ；拿不到可先暂存或降级处理。
    - 4）分组聚合 ：按 task_id 聚合一批事件，减少下游写放大。
    - 5）多路扇出写入 ：
      - 索引表user_task_index
      - 离线通道/指标通道写入
    - 6）提交位点（ack） ：主路径成功后提交消费位点。
    - 7）失败补偿 ：可重试错误指数退避；重试耗尽入 DLQ。

- 如何做存储选型的？
  
  - 场景明确：
    
    - 写：高并发持续写入（Trace事件流）
    
    - 读：读少但要快（按 task_id 拉一条任务全链路）
  
  - 查询方式：
    
    - 主要是根据task_id查询其所有事件
  
  - 评估维度：
    
    - 写入性能
    
    - 是否支持按主键/前缀快速扫描
    
    - 成本
    
    - 支持批量写入
  
  - 宽表数据库优势：
    
    - 写入模型契合“写多读少” ：Trace 事件是高并发追加写，TokaDB（类HBase宽表）对追加写与水平扩展非常友好，吞吐稳定。
    - 按主键/前缀扫描极快 ：排查最核心动作是“给我一个 task_id ，把这条链路所有事件秒级/毫秒级拉出来”。TokaDB 的 RowKey 设计（ md5(task_id) 打散热点 + task_id... 前缀扫描）能把这一查询路径做到极致。

- 怎么做索引设计的？
  
  - 查询场景：
    
    - 按 task_id 查完整链路
    
    - 按 user_id 查最近任务
  
  - 具体设计：
    
    - 主表
      
      - RowKey： `md5(task_id):task_id:tenant:business:scene:timestamp`
      - 目标：写入打散 + 按 task 前缀扫描秒级拉链路
    
    - 用户索引表（最新优先）
      
      - RowKey： `md5(user_id):user_id:reversed_timestamp`
      - Value： task_id
      - 目标：按用户查最近任务，天然按时间倒序

- 消费端如何实现退避重试与死信队列？
  
  - 消费端写入 TokaDB 下游失败。
  
  - 重试策略 ：    
    
    - 只对“可重试错误”重试（超时、限流、临时不可用）
    - 采用指数退避：例如第1次失败等 1s ，第2次 2s ，第3次 4s （可加 jitter 防止同一批同时重试）。
    - 重试上限 ：例如最多3次，避免无限占用FaaS执行时间与放大下游压力。
  
  - 重试次数耗尽仍失败
    
    - 原始消息（或携带原始 payload + error reason）投递到一个独立的 EventBus Topic（DLQ Topic）。
    - 死信队列消费者取得消息，判断error reason，若为网络等可重试问题，那么重新投入原Topic，若为不可修复问题，丢弃。

- 怎么做限流与鉴权？
  
  - 上报接口是开放的http接口的，这意味着不属于我们的业务也能通过该接口进行上报，同时我们的AI生成业务也有多个，为了能更好地记录当前核心业务的任务事件链路，进行了鉴权和限流
  
  - 每个上报请求都需要 携带tenant，bussiness、scene等字段，接收到请求需要根据配置文件对这些字段进行鉴权。不通过则直接返回。
  
  - 同时对于不同业务的限流值也会有所区别，对于当前核心业务，他的限流为600QPS左右，对于非核心则为200QPS左右。

- 你认为这个系统的核心技术有哪些？
  
  - 端到端“可定位”的数据资产 ：用 trace_id ↔ task_id 把客户端与服务端、多微服务、多地域的事件串成一条“可查询、可复盘”的任务时间线；
  
  - 为排查而生的存储与索引设计 ：TokaDB主表的 md5(task_id)+prefix scan + 用户索引表 reversed_timestamp ，把“按任务查全链路、按用户查最近任务”两类最高频排查动作做到毫秒级，这是把业务工作流映射成存储结构后的能力壁垒。
  
  - 全球化异步汇聚的工程体系 ：EventBus跨地域同步 + SG中心化FaaS消费，让任何地域的事件最终落在统一视图里，解决“数据分散、看不全、对不齐”的根因；
  
  - 从“可观测”到“可归因”的闭环 ：把查询能力封装成MCP工具接入Aime，让“查trace→找log→读代码→写结论”自动化，形成可复用的分析模板与归因流程，进一步把系统价值从“展示链路”升级到“输出结论”。

- 客户端事件与服务端事件如何打通？
  
  - - 客户端先生成 trace_id ：客户端在本地生成全局唯一的 trace_id （通常是UUID），并用它串起客户端侧所有事件（如点击、上传、前置校验等）。
    - 客户端事件批量上报 ：客户端将多个事件缓存后，通过批量HTTP接口 /tiktok/v1/aigc/trace 一次性上报 trace_items[] ；服务端入口做基础校验后把事件转发进 EventBus（客户端Topic）。
    - trace_id 透传到核心生成请求 ：客户端调用核心生成接口（如 /tiktok/v1/aigc/generate ）时，把 trace_id 放到 extra 字段一并传给服务端。
    - 服务 task_id  建立关联 ：服务端收到 generate 请求后会将自己的 task_id 与关联（代表后端生成任务实例），并把 trace_id ↔ task_id 的映射关系 落库（用于后续互查）。
    - 统一查询展示 ：后续不管用户输入 trace_id 还是 task_id ，平台都可以通过映射拿到另一侧ID，再去 TokaDB（明细）/Abase（反向索引）拉齐客户端+服务端事件，按时间线展示一条完整链路
  
  将task_id和user_id的联系放入用户索引表，abase存放task_id和track_id的映射。

- EventBus + FaaS 的组合解决了哪些问题？
  
  - - **上报与处理彻底解耦** ：业务方/客户端只负责把事件丢进 EventBus；后续解析、聚合、扇出写入 TokaDB/Abase/Hive、指标上报都在 FaaS 完成，避免业务服务背负写库与多下游复杂度。
    - **削峰填谷、抗流量突刺** ：EventBus 作为缓冲层吸收峰值写入；FaaS 的消费能力可以水平扩展，平滑落库压力，降低“高峰期直接写库把存储打爆”的风险。
    - **全球化跨地域汇聚** ：利用 EventBus 的 Geo-Replication 把 US 等地域的消息同步到 SG 主集群，FaaS 在中心地域统一消费，形成全局一致的链路视图，避免跨地域多套存储/多套查询导致“看不全、对不齐”。

- 跨地域部署的核心挑战是什么，你们如何解决？
  
  - 挑战
    
    - 事件在不同地域产生，如果各写各库，查询时需要查多个库。
    - 跨地域网络不稳定与延迟 ：长距离链路抖动会导致上报失败、重试增多、写入延迟不可控。
    - 乱序与重复 ：跨地域同步/重试会让消息到达中心的顺序与原始发生顺序不一致，甚至出现重复投递。
    - 一致性与可靠性 ：需要在“最终一致”与“可快速排查”之间做工程权衡，保证数据不丢不错、并能可解释。
  
  - 解决
    
    - EventBus Geo-Replication 做跨地域汇聚 ：US-TTP 等地域事件先进入本地MQ，再由 EventBus 自动同步到 SG 主集群，避免业务侧手写跨地域链路。
    - 中心化消费形成统一视图 ：主要 FaaS 消费集群部署在 SG，统一消费全球Topic并落到 SG 的 TokaDB/Hive，保证查询时数据完整。
    - 展示侧按时间线排序 ：即使到达乱序，最终在查询平台按 timestamp 排序还原任务时间线（减少乱序对排查的影响）。

- 为什么选择中心化消费，而不是各区域本地消费？
  
  - - 保证“全链路可见” ：AIGC一次任务的事件可能在不同地域产生（客户端、不同服务、不同机房）。中心化消费把全球事件最终汇聚到同一套落库（SG 的 TokaDB/Hive），查询时完整
    - 简化一致性与查询模型 ：如果各区域本地消费+本地落库，就会出现“**跨库拼链路、跨库按时间排序、跨库查索引**”的复杂问题；中心化消费让“一个ID→一处查全链路”成立，平台实现和运维更简单。
    - 降低运维与成本 ：每个地域都部署一套FaaS消费+TokaDB+索引+离线链路，资源与人力成本线性增长；中心化把复杂度集中在一个主集群，规模效应更明显。
    - 更易做全局监控与归因 ：全局指标、失败模式统计、Aime自动归因需要全量数据；中心化消费天然利于做全局看板与离线分析训练。

- 这套架构在高峰期的瓶颈在哪里？
  
  - - 入口层（HTTP批量上报 / SDK上报） ：高峰时请求量暴涨会先打满网关/服务端线程池；如果批量策略不合理（包太小、频率太高），QPS会被放大。
    
    - EventBus（MQ吞吐与跨地域同步） ：Topic写入吞吐、分区热点、Geo-Replication带宽/延迟都会成为瓶颈；一旦同步变慢，中心地域会出现消费落后与积压。
    
    - FaaS消费层（并发与冷启动） ：中心化消费在SG承接全球流量，高峰时容易出现函数并发上限、冷启动抖动、批量拉取/解析CPU成为瓶颈，进而导致MQ积压。
    
    - 下游存储写入（TokaDB/索引表/Abase） ：这是最常见的硬瓶颈：多路写入放大写QPS；RowKey设计不当会形成热点Region；索引表写入与主表写入叠加导致写放大。

- 客户端批量上报的触发策略是什么？
  
  - - 数量阈值触发 ：本地队列累计到10条数就立刻打包上报（避免无限堆积、控制单包成本）。
    - 时间阈值触发 ：即使没到数量阈值，也会5min定时flush（保证链路可见性与延迟上限）。
    - 生命周期触发 ：App进入后台/即将被挂起/网络切换等场景触发flush（降低丢失概率）。

- 客户端断网或App退出时如何保障数据不丢？
  
  - - 本地缓存队列 ：客户端不是每条事件立刻发HTTP，而是先写入本地缓存（队列/缓冲区），再批量上报；断网时继续入队，网络恢复后再统一flush。
    - 批量上报 + 重试 ：通过 /tiktok/v1/aigc/trace 批量提交 trace_items[] ，失败时不丢弃而是进入重试流程（通常带退避），减少弱网下的丢包概率。

- HTTP入口的参数校验机制如何设计？
  
  - 放在网关/中间件层做第一道拦截 ：在 /tiktok/v1/aigc/trace 这种统一入口，尽早做校验，避免脏数据进入 MQ 与下游存储。
  
  - **白名单校验**（强约束字段） ：从**配置中心（TCC/Apollo）加载允许接入的 tenant / business_scene / scene / level 白名单**；请求到达时逐项校验，不在白名单直接拒绝并返回可机读错误码（比如把 business_scene 的非法单独定义错误码）。
  
  - 结构与必填校验 ：校验 trace_items[] 非空；每个 item 必须有 event_name / timestamp / trace_id(或task_id) 等核心字段； timestamp 合法范围（不能未来过大、不能过旧离谱）。

- 跨地域同步是否会引入乱序，如何处理乱序？
  
  - 跨地域同步（Geo-Replication）会受到网络延迟、重试、分区/批量转发、以及不同地域产生事件的时间差影响，导致消息到达中心（SG）的顺序和原始发生顺序不一致。
  
  - - 以事件时间为准，而不是到达时间 ：事件Schema里带 timestamp ；存储侧的RowKey也包含时间戳，便于按时间扫描与排序。
    - 中心化落库统一视图 ：所有地域事件最终汇聚到中心消费并落到同一套存储，避免“跨库拼接带来的更严重乱序/缺失”。

- 消息积压时的系统行为是什么？
  
  - - 扩容消费能力 ：提升FaaS并发、增加消费实例，优先把 backlog 降下来（这是最直接的手段）。
    
    - 批量聚合与批写 ：消费端按 task_id 聚合后批量写入，减少数据库请求次数，提升单位吞吐。

- “写多读少”场景对存储提出了哪些要求？
  
  - - 高吞吐追加写 ：能承受持续高QPS写入，最好是追加/顺序写友好，避免复杂索引导致写入成本爆炸。
    - 水平扩展能力 ：数据量与写QPS随业务增长，需要能通过加节点扩容，而不是单机/单表瓶颈。
    - 热点规避能力 ：写入键容易集中，存储必须支持通过RowKey设计/分区策略打散热点，否则某个分区会被打爆。
    - 主键/前缀查询极快且稳定 ：虽然读少，但读往往是“排查关键路径”，要求按 task_id 这类Key能毫秒级拉到完整明细（前缀扫描/范围扫描能力强）。
    - 成本可控 ：大量数据长时间沉淀，要求单位存储成本、写入成本可控；不能为了少量读取把倒排索引等高成本结构全建起来。

- TokaDB的短板是什么，如何规避？
  
  - - 对动态字段检索能力弱 ： context(map[string]string) 很灵活，但在宽表里几乎无法对 context 内任意key做高效条件查询（没有倒排索引/高效二级索引体系），只能按RowKey查或全表/大范围扫。
    - 查询形态受限 ：适合“**按主键/前缀扫描**”，不适合“按任意维度筛选 + 聚合统计”（例如按 moderation_reason、model_name 做复杂分析）。
    - RowKey 设计决定一切 ：一旦RowKey设计不当就会出现热点、scan范围过大、Region负载不均等问题；且后期改RowKey迁移成本高。
    - 多表/多索引一致性成本 ：为了支持更多查询（user索引、event索引），需要**额外索引表**，写放大与一致性治理复杂度上升。

- - 明确在线主路径 ：把在线排查定位为“按 task_id 查全链路、按 user_id 查最近任务”，这两类都能用RowKey前缀扫描做到极致。 
  - 热点打散 ：主表RowKey加 md5(task_id) 前缀，避免同一task写热点集中到某个Region。
  - 二级索引表解决非主键查询 ：建立 user_task_index 表，用 md5(user_id)+reversed_timestamp 支持按用户查最近任务。
  - 字段提升（把常用动态字段变固定列） ：把高频需要筛选的字段从 context 提升为固定列（并可建立索引），把 context 保持为payload。
  - 在线/离线分层 ：复杂维度查询与归因分析走 Hive 离线仓；TokaDB只承担实时明细查询。

- Abase 反向索引的必要性是什么？
  
  - - 打通 trace_id ↔ task_id 的快速互查 ：客户端侧更容易拿到 trace_id ，服务端排查更常用 task_id 。Abase 存一份映射（反向索引），让平台能“拿任一ID都能秒级定位到另一侧ID”，再去 TokaDB 拉全链路明细。
    
    - 避免在 TokaDB 做非主键查询/大范围扫描 ：TokaDB主表的主路径是按 task_id 前缀扫事件；如果用户只给了 trace_id ，没有索引就只能做昂贵的扫描或额外建复杂索引。Abase 用 KV 查映射是 O(1) 的。
    
    - 降低跨端/跨系统接入耦合 ：服务端生成 task_id ，客户端生成 trace_id ，两边都不需要改各自ID体系；只要在消费/处理链路里补一条映射写入，就能把两端链路统一到一个查询入口。

- 主表RowKey的设计逻辑是什么？
  
  - - RowKey 形态 ： md5(task_id) : task_id : tenant : business : scene : timestamp
    - 设计意图拆解 ：
    - md5(task_id) 前缀 ：把同一个业务在高峰期的大量写入打散到不同 Region/分片，避免“某类task集中落到同一分区”导致热点与写入抖动。
    - task_id:tenant:business:scene 作为核心前缀 ：保证查询时可以用这段做 Prefix Scan ，一次范围扫描拿到该任务所有事件（这是排查的主路径）。
    - timestamp 放在末尾 ：保证同一 task 下事件按时间字典序排列（物理上更接近时间线），并确保 RowKey 唯一（同一任务多事件不会互相覆盖）。

- md5(task_id)前缀具体解决了什么问题？
  
  - 避免写热点：宽表按 RowKey 的字典序把数据分配到不同 Region/分片。如果 RowKey 一开始是 task_id 或 timestamp 这类“相近/递增”的前缀，高峰期大量写会集中落到少数几个 Region

- reversed_timestamp 如何实现“最新任务优先”？
  
  - - 反转时间戳定义：reversed_timestamp = Long.MAX_VALUE − timestamp
    - 字典序效果：越新的 timestamp 值越大，反转后越小；在宽表按 RowKey 升序扫描时，最新记录会排在最前面
    - 索引行键：user_task_index 采用 md5(user_id):user_id:reversed_timestamp，按 user 前缀做 Prefix Scan 可天然得到“最新任务优先”的结果

- 如果新增一个按event_name查询需求，如何设计索引？
  
  - md5(event_name) : event_name : tenant : business : scene : reversed_timestamp

- TokaDB中的“一行数据“包含什么？
  
  - task_id/event_name/level/timestamp/extra

- 这个系统最大的不足是什么，你如何改进？
  
  - 查询维度受限：主表偏向“按 task_id 前缀扫”，对 context 等动态字段几乎不可检索，导致运维/归因在复杂筛选场景下体验弱。

## CA

我做的是一个面向业务 Demo 快速搭建的多 Agent 应用，让PM可以不借助研发，只需要在系统中输入需求，就能以最快时间体验到他想要的AI链路，PM可以直接对生成的代码运行测试。

这个项目的前景是，提前已经实现了多个AI原子能力并封装到服务器上，可以通过http请求进行调用，当时的想法是给出一些对这些能力进行组装调用的代码实例，PM如果有需要，可以参考这些代码自己写出自己想要的AI链路，但由于这样对于PM的代码能力还是有一定要求，他们进行组装十分困难，故而设计了这样一个Agent。

输入是PM对于demo应用的需求，比如（根据用户意图进行图片修改、生成等），输出是一份可运行代码，可以直接在系统中运行，可以对其代码中可调整参数（如测试用例 图片输入、文本输入、或者是底层模型的prompt）进行调整。 

系统分为Planner Agent、Memory Agent、Code Agent：Planner 负责意图理解和任务调度，Memory 负责历史版本与上下文检索，Code Agent 负责 TODO 驱动的代码生成、测试和修复闭环。

- 链路和架构
  
  - Planner Agent，负责调度意图理解得到具体需求点、链路规划 & 原子能力选择（图片生成、修改，视频生成、修改、抽帧、跟拍，合规检测，语言翻译）、Memory、代码Agent
    
    - 在Planner中先对用户输入进行简要理解，判断其意图是新建、修改、其他（单纯问答），如果是修改那么调用Memory获取信息。
    
    - Planner再根据Memory返回的信息（会返回上下文文件路径）与用户输入进行精细理解，得到具体需求点，判断是否需要ask_user。
    
    - Planner Agent 会根据上下文判断是否需要进入codeagent。给出需求点是什么。
  
  - Memory Agent：进行上下文的检索，使用文件类工具，获取有关上下文
  
  - CodeAgent，负责规划（写入todo list）、代码生成、代码测试、debug、生成代码评估，todo write工具。
  
  - 一轮对话完成，将版本信息写入文件，目录结构为 `sessions/{sessionId}/{versionId+变更信息}/`目录下 结束时的`todo.md`，版本摘要信息文件（用户输入、意图，执行步骤，结果，错误原因，变更摘要），详细信息+生成代码（选择工具、链路信息以及摘要中各部分的详细内容）。
  
  - 代码生成成功，页面出现按钮，提问用户是否成功解决。确认成功则由LLM以固定格式总结本轮的对话，写入文件，作为rag知识来源。

- 工具
  
  - 人机交互：当用户的输入约束过少，LLM判断需要用户输入额外信息，将会输出一个工具调用信息 - 固定格式的json - 包含想要询问的问题（ask_user），系统收到该消息后，向前端发送一个特定状态码+payload，然后将该agent保存状态并中断，前端根据payload向用户询问，用户输入后，前端向后端特定接口发送信息，后端重新恢复agent。
  
  - 读写文件内容、读文件名、读目录名、创建目录，read_file / write_file / list_files / list_dirs / mkdir
  
  - docker代码测试：传代码文本
  
  - todo write：只有code agent有，进行增量更新，不删除已完成的todo，将todo转为md传给LLM

- system prompt
  
  - 角色与场景、目标、输入说明（对user prompt的字段进行解析）、工作流程、输出格式、强约束。
  
  - 只对一句话进行增删改，避免出现问题不知道是哪句话引起的。

- user prompt
  
  - 以json格式组织，对应system prompt中的字段说明。

- 上下文工程
  
  - Planner Agent：当前用户需求，可用原子能力信息，产生的链路，意图理解结果，由Memory Agent获取到的历史版本有关信息。
  
  - Memory Agent：根据用户输入获取上下文
  
  - code agent： 在agent中产生的信息，以及主agent 获取到的版本信息（包括代码、变更记录）、工具错误信息。

- rag
  
  - - 需求（本轮对话的需求）+ 需求类型
    - Todo List简化版
    - 关键决策点（为什么这么做）
    - 失败与修复回路（错误摘要→修复策略→验证→预防）
  
  - 切片策略
    
    - 每个 chunk 头部 带 需求摘要、需求类型、使用能力 的极短头部，避免单 chunk 命中后看不懂。
    
    - 决策点/失败回路必须“按条切” ，保证遇到某种错误时可以准确找到解决策略。
    
    - todo整切
  
  - 在用户点击确认有用之后将如上格式、策略 分为多个文件存储，决策点、失败与修复回路需要按条（也就是按段落）切成chunk再存入数据库，其他文件直接向量化即可。

- 你如何评估检索召回内容与用户问题的匹配度，有没有做相关的效果评估？
  
  - 通过对比加不加RAG代码生成的效果来评估
    
    - 成功率： 从75% 到 90%左右
    
    - debug 次数： 从2 ～ 3次 到 最多1次
    
    - 任务耗时： 根据debug次数而减少， 从 12分钟 到 6 分钟左右
  
  - 在引入RAG到系统之前，进行了单独实验评测（构建两个常见场景的query）提前标注在知识库应该被召回的片段，检测chunk策略是否能够召回有用知识，召回的是否都是有关的？有无错误引导召回？

- RAG 知识库文档结构是什么？为什么这样设计？
  
  - 需求（本路对话的摘要）+ 需求类型
  
  - Todo List简化版
  
  - 关键决策点（为什么这么做）
  
  - 失败与修复回路（错误摘要→修复策略→验证→预防）

        让Agent聚焦于步骤、决策、修复方法，而不是其他

- 你的 Agent 循环一般多少步可以完成任务，有没有出现过达到最大步数仍无法完成的情况？
  
  - 简单任务：Planner 4轮（简单意图+完整需求理解+原子能力选择+Code）、Memory无、Code 3轮（无需debug）
  
  - 复杂任务：Planner 5轮（再加上memory）、Memory 2轮（版本目录名初筛 + 摘要精筛）、Code 12轮（debug最多3次）
  
  - 遇到过，在Code进行debug修复后又因为错误信息没有及时清理（之前是使用字段对LLM进行约束，如果有code字段、有错误信息字段，那么就执行debug，而debug后没有及时对错误信息字段清空）导致继续重复debug，后续修改了debug环节的逻辑、限制debug最大次数、在Code中增加TODO List明确已经执行过的历史步骤，避免重复调用debug工具或运行工具。

- 为什么不直接用其他codeAgent？比如ClaudeCode
  
  - - api文档全部发给AI，可能会造成上下文窗口拥挤，导致AI能力下滑。
    
    - 其他的生成代码后需要自己进行实际测试，如果报错还要将错误信息发回给AI。

- 为什么需要多 Agent，而不是一个大 Agent？
  
  - 多Agent需要的上下文、工具不同；拆开后每个 Agent 的 System Prompt 更短更清晰，行为更稳定。

- 多Agent协作流程
  
  - Planner 根据 用户输入 判断本任务是与历史对话相关，如果相关，那么调用Memory Agent进行完整的上下文获取。得到历史上下文后，Planner 进一步理解用户需求。选择使用Code Agent 或 其他工具。

- Agent之间怎么通信？
  
  - 每一个会话有自己的state状态， 三个agent分别读取写入其中的某些字段
  
  - 当需要调用子Agent时，planner做出下一节点的决策，同时将要传递的信息写入state中，并通过调度节点读取决策跳转到具体子agent，子agent任务完成后通过state返回结果。

- 主 Agent 如何判断“是否进入 CodeAgent”？
  
  - 根据意图理解结果判断是否需要对代码/文件做真实变更，否则直接在主 Agent 内完成解释与规划。

- 未来可以怎么进一步划分子agent？
  
  - - Orchestrator（主 Agent） ：意图理解、路由、ask_user、总评估
    - CodeAgent ：TODO→改代码→Docker 验证→Debug 闭环
    - EvaluationAgent ：基于执行报告评估质量、决定是否继续
    - MemoryAgent：版本摘要/日志压缩/写 RAG 文档
    - ToolingAgent（可选） ：能力网关调用、外部服务编排

- token消耗量怎么样？不同类型的任务不同
  
  - 无需进入代码agent - 大概20k tokens
  
  - 进入代码agent，但只需要执行一轮无需debug即可生成完成 - 大概100k
  
  - 需要进行多轮debug，重复出错、修正 - 大概350k 

- 怎么减少token消耗？
  
  - agent之间隔离，避免重复出错导致的token消耗
  
  - 三段式加载、压缩
  
  - 对工具按需注入，原子能力只传摘要、对真正要用的才传完整描述。
  
  - 简单意图分类、规划使用较低成本模型，代码生成使用前沿模型。

- TODO具体流程
  
  - 代码agent拿到需求点后，先生成执行计划，写入TODO List，以结构体数组的格式
  
  - 代码agent，根据TODO（转为md格式）以及其他上下文，决定下一步骤并执行，同时改变todo里面的状态
  
  - 如果代码agent判断需要额外执行其他步骤（必须debug之后重新运行测试），那么调用工具对Todo List插入一条

- 为什么要用 Tool Calling，而不是让 LLM 直接输出文本？
  
  - 纯文本输出很可能出现格式错乱问题，或者文本不对问题，导致无法解析到真实工具。Tool Calling 在 LLM训练层面让它在决定调用工具形成固定输出。

- 你是如何约定并约束大模型进行工具调用的？
  
  - 使用标准的Tool Schema结构定义工具，通过LLM的tools字段传给他工具信息
  
  - 不在system prompt中规定工具调用输出格式，只需要告诉它必要时选择工具调用即可，通过 LLM 的 tool_call字段读取工具调用信息。
  
  - 系统解析tool_call中固定的格式信息，进行实际工具调用。

- LLM 如何更新 TODOList？
  
  - 给出几个工具，全量写入、插入、状态更改。
  
  - 标记已完成操作，设置某一项的状态-in_progress、completed

- “暂停—提问—恢复”在后端怎么实现？
  
  - 当用户的输入约束过少，LLM判断需要用户输入额外信息，将会输出一个工具调用信息 - 固定格式的json-包含想要询问的问题（ask_user），系统收到该消息后，向前端发送一个特定状态码（202）+payload（waitting_for_user，问题） sse事件，然后将该agent保存状态并中断，前端根据payload向用户询问，用户输入后，前端向后端特定接口发送信息，后端重新恢复agent。

- 为什么不让 agent 阻塞等待用户输入？
  
  - 线程资源会被长时间浪费，长时间连接可能被断开

- 工具多了会 token 爆炸，你怎么处理？
  
  - 分类，LLM判断需要使用哪类工具再暴露。
  
  - Skills，特定场景才暴露tool schema

- 为什么选择 Docker 执行，而不是本地执行？
  
  - 安全隔离、资源可控、清理成本低

- Docker执行过程
  
  - create 创建容器、固定配置 → start -a 启动并采集输出 → inspect 取结束状态 → rm -f 清理容器

- 如何处理超时？
  
  - 对docker执行命令定时

- 三段式加载流程
  
  - 触发时机：判断需要历史版本上下文时，通过Memory Agent进行加载
  
  - 初筛（文件名/索引级）
    
    - 系统先把 版本目录名 列表 （例如 v0012__fix-login__err-401 ）提供给 LLM
    - LLM 根据任务意图选 TopN 候选（5 个）
  
  - 摘要（summary 级）
    
    - 仅加载 TopN 版本的 summary.json （包含意图、步骤轨迹、变更摘要、错误原因、执行结果）
    - LLM 进一步筛选出 TopK （1个）
  
  - 全文（full context 级）
    
    - 仅对 TopK 版本加载必要的“全文内容”
      - 代码文件（按需读）
      - 详细执行日志

- 三段式压缩流程
  
  - 一轮对话完成后进行写入，将版本信息写入文件
  
  - 目录结构为 `sessions/{sessionId}/{versionId+变更信息}/`
  
  - 在该目录下，写入结束时的`todo.md`，摘要信息文件（用户输入、意图，执行步骤，结果，错误原因，变更摘要），详细信息+生成代码（选择工具、链路信息以及摘要中各部分的详细内容）。

- 主 Agent 与 CodeAgent 各自上下文里应该有什么？
  
  - - 主Agent：当前用户需求，产生的链路，意图理解结果，需求点
    
    - code agent： 在agent中产生的信息，以及主agent 获取到的版本信息（包括代码、变更记录）、工具错误信息。

- 怎么版本回退？
  
  - 在前端显示的对话列表中，有一个按钮，点击后可确认回退到该版本，后端会直接将该版本之后的所有版本文件删除，用户重新开始对话，不会再使用到之前的历史上下文。并且前端将后面的信息清除。

- 什么时候会用到RAG？
  
  - 代码agent规划（生成todolist）阶段，检索相似成功路径，召回Top3
  
  - 修复阶段：Docker 执行失败时，用错误摘要检索 fixloop，召回Top3

- 为什么要做流程型 RAG，而不是代码型 RAG？
  
  - 代码不需要参考，也没有参考意义，降低token
  
  - Agent遇到某些问题时该如何处理才是有意义的。可以减少执行轮数。

- 向量检索召回的单次耗时是多久
  
  - 在300ms左右

- 如何保证用户体验稳定（少澄清、少失败、少等待）？
  
  - 一次性澄清策略 ：主 Agent 不零散追问，而是根据任务类型生成“最小必要信息表单”（ask_user fields），一次补齐关键约束。
  
  - 流式反馈 ：用 SSE 推送状态与关键日志（而不是沉默等 2 分钟），让用户感知“在推进”。

- User Prompt 为什么要 JSON 化？有什么好处？
  
  - 让模型更少误解、更少遗漏、更容易稳定执行
  
  - 可以规定字段优先级，模型更容易按规则理解，而不是凭语气猜。
