### 你对 Agent 的理解是什么，它包含哪些核心模块？

- 一个基于 LLM 的智能体（Agent）是一个能够自主理解环境、进行规划决策、并执行行动以达成特定目标的计算系统。其核心特征是利用一个**大型语言模型（LLM）作为其“大脑”或“中央处理器”**，来进行复杂的推理和决策。它能主动地、持续地与环境或工具交互，直到完成任务。
- 主要模块：
  - LLM - 大脑
  - 规划能力
  - 记忆模块
  - 工具使用能力

### 如何在一个给定的场景中进行agent的总体设计？

- 明确输入与输出，明确agent所处场景，明确拥有的外部信息（比如获取信息的接口）（开阔思维，可以对拥有的外部信息进行扩展）

- 对总体任务进行分解，并进行耦合性分析，较复杂的子任务建议单独一个Agent

- 判断用单Agent还是multi-Agent，如果子任务复杂 或 可以通过并行子任务做到提升效率，那么使用multi-Agent。

- 判断Agent的流程是相对固定还是发散，如果流程较固定那么可以使用`Plan-and- Execute`，可以减少token消耗以及加快速度。使用ReAct则会更加动态化。

- 明确哪些信息是必须要从Tool调用中获取的，而不能让LLM瞎猜，设定合理的Tool Use。（比如计算、判断数据有效、用户信息异常的检测、严格的规则校验）。

- 设计记忆模块，是否需要保存用户的最近会话？是否需要根据动态的知识库来获取某些强制性规则？

- 对于针对输出内容比较严格的场景，可以在输出内容时再次进行一次强制规则性判断，避免LLM由于幻觉而绕过了某些强制性规定。

### Agent 的设计范式

- ReAct  -  `问题 -> 思考 -> 行动 -> 观察 -> 思考 (循环) -> 最终答案`。

- Plan-and-Execute (规划-执行)： 将大目标拆解为一份完整的步骤清单， 逐一执行清单任务，并根据结果更新计划。

- Reflexion：包含三个角色：**Actor**（尝试解决）、**Evaluator**（打分/找茬）、**Self-Reflection**（根据反馈总结教训）。

- 多Agent协作

### 对 MCP、Function Call、A2A 分别怎么理解的？

- Function Call让LLM拥有调用工具的能力，我们需要给LLM工具的Json描述信息，LLM就能在需要使用工具的时候，返回固定格式

- MCP - 模型上下文协议：定义了一个通用的标准，让数据源（上下文）和 AI 应用可以无缝对接，解耦上下文数据、工具开发 与 Agent开发，同一个工具可以提供给多个agent使用，不需要重写或复制

- A2A：为了实现多个异构Agent之间的通信和协作，它试图定义一套**通用的标准、协议和语言**，使得由不同开发者、为了不同目标而构建的Agent们，能够相互发现、理解和交互。

### Skills与Tool Call的区别

- Tool Call只简单地让LLM能够调用外部工具，没有对Tool该怎么用好、出现问题该怎么做进行说明。
- Skills提供领域专业知识，可能包含不止一个Tool，也可能拥有asset资源包，比如一个pdf Skill，里面就可能包含一个用于写PDF的tool，一个PDF标准格式的模板，并且内置prompt用于告诉智能体在特定场景下"如何组合使用这些工具"

### Skills与MCP的区别

- **MCP 的职责**：提供标准化的访问接口，让智能体能够"够得着"外部世界的数据和工具
- **Skills 的职责**：是对一个特定能力逻辑的**封装**。一个 Skill 通常包含：一段 Prompt 指令、一个或多个工具调用（Tool Calls）、以及处理结果的逻辑。提供领域专业知识，告诉智能体在特定场景下"如何组合使用这些工具"
- 如果使用mcp将所有工具一股脑发给LLM，会很快的导致上下文爆炸，而使用Skills，LLM则会在特定适用场景才会披露里面的Tool Schema。

### 使用AI工具的心得（最佳实践）

- 用途：
  
  - 遇到难理解的问题、概念时，让AI以通俗的方式进行举例解释
  
  - 对论文进行润色，让其更加连贯通顺
  
  - 进行研究课题灵感的获取，让AI基于当前方向，给出多个可能的扩展点。
  
  - 进行项目数据流图的理解，快速了解项目的大体流程，更快融入业务。
  
  - 使用vibe coding完成项目的开发，单元测试的编写

- 最佳实践：
  
  - 创建Agent.md 或 Claude.md文件，记录技术栈、编码风格、强制约束等
  
  - 让claude先对项目结构、层级进行理解，让AI对某个文件实现了什么功能有个大概的了解
  
  - 给出生成某个模块的任务指令，并让Claude输出一份执行计划（一般来说都是先 生成框架、再对具体功能进行逐步实现）
  
  - 审阅执行计划，有问题则进行修改
  
  - 将计划步骤逐个让claude实现
  
  - 全部完成后，让claude编写单元测试并执行。

- 为什么不用Plan模式而是输出一份计划文本？
  
  - 让claude执行一个较长的计划，可能因为前面的某一步骤出错，而导致后续所有环节都出问题，最后的结果肯定是错误的结果。
  
  - 将计划拆开单独发给claude，每一步计划之后都可以审阅代码有无问题？可以减少一次任务的上下文污染

### 是否了解OpenClaw以及了解它的实现原理？

- 是一个负责编排大模型、记忆系统、外部工具和通讯平台的**代理框架**。用户可以直接在通讯平台上给他排发任务，它将通过调用底层LLM，进行意图理解、逻辑推理，决定应该怎么完成任务。

- 实现原理：
  
  - 通讯网关层：使用websocket与用户在通讯平台上保持实时通信。
  
  - 推理与代理层：进行逻辑推理，决定子agent或工具的调用。
  
  - 记忆管理：上下文压缩到文件中
  
  - 技能执行层：通过安装Skills插件，可以实际完成任务，比如搜索网页，执行shell命令，完成屏幕点击事件。

## ----LLM相关----

### 自注意力机制是如何工作的？

- 对于输入序列中的每一个词元（token）的嵌入向量，我们通过乘以三个可学习的权重矩阵，分别生成三个向量：查询向量（Query, Q）、键向量（Key, K）和值向量（Value, V）。
  
  - **Query (Q):** 代表当前词元为了更好地理解自己，需要去“查询”序列中其他词元的信息。
  
  - **Key (K):** 代表序列中每个词元所“携带”的，可以被查询的信息标签。
  
  - **Value (V):** 代表序列中每个词元实际包含的深层含义。

- 计算当前词元的**Q与其他所有词元的K**的点积。这个分数衡量了两者之间的相关性。

- 计算出的分数除以一个缩放因子

- 将缩放后的分数通过一个Softmax函数，使其转换为一组总和为1的概率分布。

- 得到的**注意力权重与每个词元对应的V向量相乘并求和**，得到最终的自注意力层输出。

### 什么是位置编码？

- 位置编码（Positional Encoding, PE）是一个**与词嵌入维度相同的向量**，其目的是向模型注入关于**词元在输入序列中绝对或相对位置**的信息。它会与**词元的词嵌入（Token Embedding）相加**，然后一同输入到Transformer的底层。

- 自注意力机制，在计算时处理的是一个集合而非序列。它本身不包含任何关于词元顺序的信息，语序不同但它的计算结果是一样的。

### 什么是ROPE

- 通过向量旋转的方式，将**绝对位置信息编码到Query和Key向量**中，从而使得模型在计算注意力分数时，能够自然地利用相对位置信息

### 几种常见的 LLM 架构

- Encoder-Only 架构
  - 双向自注意力机制。在处理序列中的任何一个词元时，模型都可以同时关注到它左边和右边的所有词元。这使得模型能够获得非常丰富的上下文表示。
  - 适合自然语言理解任务：情感分析
- Decoder-Only 架构
  - 单向（因果）自注意力机制 (Causal Self-Attention)。在**预测第 `t` 个词元时，模型只能关注到位置 `1` 到 `t-1` 的词元**，不能看到未来的信息。这种自回归的特性天然适合生成任务。
  - 适合自然语言生成任务：代码生成、聊天机器人
- Encoder-Decoder
  - Encoder部分使用**双向注意力**来编码整个输入序列，形成一个全面的上下文表示。Decoder部分在生成输出时，一方面使用**单向注意力**处理已生成的序列，另一方面通过**交叉注意力**机制来关注Encoder的输出，确保生成内容与输入相关。
  - 适合 自然语言翻译

### 什么是词元化

- 词元化是**将原始的文本字符串分解成一个个独立的单元**（称为“词元”或“token”），并**将这些词元映射到唯一的整数ID的过程**。

### 什么是Scaling Laws？

- 一条经验规律：揭示了大型语言模型的性能与三个关键资源要素——**模型参数规模（N）**、**训练数据集大小 D**和**训练所用的计算量 C**——之间存在着可预测的**幂律关系**

### 涌现能力是什么？

- 当模型规模（包括参数量、训练数据和计算量）达到某个临界点（百亿到前亿）后，突然出现并显著超越随机水平的能力（思维链、few-shot）

### RLHF流程的三个核心阶段

- 背景：一个LLM经过训练后，得到的是一个只会续写的机器，你提出问题，它之后接着这个问题进行续写，而不是回答

- 步骤
  
  1. **有监督微调**：给出一个高质量的、由人工编写或筛选的指令数据集。数据格式通常是（指令 Prompt, 理想回答 Response）。对LLM进行微调，让LLM能够遵循人类指令。
  2. **训练奖励模型**
     - 输入一个人类偏好比较数据集。让人类对微调后的模型的多个输出进行哪一个更好的比较。
     - 最终得到一个奖励模型，这个模型能够对微调后的模型的输出进行评分。
  3. **近端策略优化** （PPO）：使用奖励模型 通过强化学习 对 微调的模型参数进行调整，让LLM能够输出更符合人类期望的答案。

### DPO的核心思想

- DPO是一种更简单、更稳定的语言模型偏好对齐方法，其核心思想是 **绕过** 显式的**奖励模型建模和复杂的强化学习训练**过程，直接**利用偏好数据来优化语言模型**。
- DPO将RLHF这个“**先学习奖励，再用RL优化**”的两阶段问题，直接转换成了一个等价的“**直接用偏好数据进行监督学习**”的一阶段问题。

## ----前沿AI----

### Mem0   -   Memory Zero

核心是**优化Agent的长期记忆**

- 输入与解析 ：接收用户与 AI 的历史对话消息，同时绑定上下文标识（如 user_id 、 agent_id 或 run_id ）以便**实现数据隔离**。
- 事实提取：它不会将整个对话原文存入数据库，而是通过调用 LLM（配合定制的 Prompt）**从对话中提取出结构化的、独立的事实**（例如：“用户喜欢吃奶酪披萨”、“用户的名字叫 John”）。
- 旧记忆检索 ：对提取出的每个“新事实”，将其转化为向量表示（Embedding），并**在底层 Vector Store 中搜索最相似的已有记忆**。
- 智能记忆决策：将“新事实”与“相似的旧记忆”同时提供给 LLM，**让 LLM 扮演记忆管理者的角色，决定对旧记忆执行哪种操作**：
  - ADD ：完全新的信息，生成新 ID 并添加。
  - UPDATE ：同一主题的信息发生了变化或变得更丰富，保持 ID 不变**更新内容**。
  - DELETE ：新信息与旧信息**产生矛盾**（如：“我现在不喜欢吃披萨了”），则**删除旧记忆**。
  - NONE ：信息已存在且无新意，不进行任何操作。
- 持久化 ：根据 LLM 的决策，将最终数据同步更新到向量数据库（Vector Store）、图数据库（Graph Store，如果开启的话），并在 SQLite 中记录操作历史以供溯源。

解决问题：

- 传统的 RAG系统只管“存”和“查”。如果用户昨天说“我喜欢吃牛肉”，今天说“我最近改吃素了”，RAG 会把这两句话都检索出来塞给大模型，导致Agent不知道到底该听哪句。而Mem会对历史记忆进行增删改查。

- Mem0引入了知识图谱。它在保存时会提取出实体和关系： [User] -[WORKS_AT]-> [Apple] <-[CEO_OF]- [Tim Cook] 。当 Agent 回答问题时，它可以**沿着图谱的边进行多跳推理**，从而回答出必须整合多段历史才能得出答案的复杂逻辑问题。

- 通过agent_id对不同agent的记忆进行隔离。

### PaperIndex

用于把长 PDF 或 Markdown 文档转成可推理的“层级树结构”（类似强化版目录），再用于检索和问答场景。

核心输出是一个带层级节点、起止页、可选摘要/文档描述的结构化 JSON，能让 LLM 按“章节树”进行树搜索式检索。
