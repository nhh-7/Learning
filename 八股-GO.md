### Goroutine与线程的区别

1. 内存消耗
   
   线程： 创建一个系统级线程通常需要 1MB - 2MB 的栈内存。
   Goroutine： 初始栈空间仅需 2KB 左右。它会根据需要动态地伸缩（最大可达 1GB）

2. 调度方式 (M:N 模型)
   线程： 由操作系统内核调度。当线程切换时，CPU 需要从**用户态切换到内核态**，保存大量寄存器的值等，这种“上下文切换”开销较大。
   Goroutine： 由 Go 运行时 自行调度。切换一个协程只需要保存三个寄存器：程序计数器（PC）、堆栈指针（SP）和 BP。且有高效的GMP调度模型，**M 个协程对应 N 个系统线程**。

3. 创建与销毁
   线程： 涉及系统调用，开销高。
   Goroutine： 纯用户态操作，创建和销毁的速度极快，几乎可以忽略不计。

### go实现继承

**实现原理**：把“父类”作为一个没有名字的字段嵌入到“子类”中，子类就能直接调用父类的属性和方法。

```go
    }

    fmt.Println(d.Name) 
    d.Eat() 
}
```

### make 和 new 的区别？

**`make`**：

- 用于初始化并分配内存，只能用于创建 `slice`、`map` 和 `channel` 三种类型。
- 返回的是初始化后的数据结构，而不是指针。

**`new`**：

- 分配内存，并把内存**清零**，返回一个**指针**（`*T`）。
- 可以用于任何类型的内存分配。

### 数组 (Array) vs 切片 (Slice)

- **数组**：
  - **值传递**：赋值或传参时会**拷贝整个数组**。
  - **死板**：长度是类型的一部分（`[3]int` 和 `[4]int` 是不同类型），**定义后长度不可变**。
- **切片**：
  - **引用传递**：本质是一个结构体，内部包含指针。拷贝时代价很小。
  - **灵活**：**长度可变**，底层基于数组。它有三个属性：指针（指向底层数组）、长度（len）、容量（cap）。

### `for range` 的地址陷阱（版本差异）

- **Go 1.22 之前**：在循环中，迭代变量（如 `for i, v := range` 中的 `v`）是**复用**的。也就是说，整个循环过程中 `v` 的**内存地址不变**，只是值在变。如果你在循环里启动协程引用了 `&v`，所有协程最后都会拿到同一个值（通常是最后一个元素）。
- **Go 1.22 及之后**：修复了这个问题。每次循环迭代都会**创建一个新的变量**，地址是不同的，不再共享。

### 字符串拼接哪个最快？

- **推荐**：`strings.Builder` 
  - **原理**：内部维护一个 `[]byte`。它通过 `unsafe` 包将 `[]byte` 直接转换为 `string`，**避免了最后一步的内存拷贝**。
- **一般**：`bytes.Buffer`（比 Builder 稍慢，因为最后`[]byte`转 `string` 时有一次内存拷贝）。
- **最慢**：`+` 号拼接。因为它每次拼接都会创建一个新的字符串对象，对原来的字符串遍历放入到该对象中，频繁分配内存。
- **慢但灵活**：`fmt.Sprintf`（涉及反射有性能损耗和复杂的格式化逻辑）。

### `defer`（延迟执行）

- **执行顺序**：**栈**结构，后进先出（后定义的 defer 先执行）。
- **时机**：函数执行完毕后、返回前执行。即使发生 panic 也会执行。
- **场景**：资源释放（关闭文件、解锁 mutex、关闭数据库连接）。

   在同一个源文件中，init 函数会按照它们在代码中**出现的 先后顺序** 依次执行。

### Go语言的多返回值是如何实现的？

Go语言的多返回值是通过**在函数调用栈帧上预留空间**并进行**值复制**来实现的。在函数调用发生时，Go编译器会计算出函数所有返回值的总大小。在为该函数创建栈帧时，就会**在调用方（caller）的栈帧**上，为这些返回值预留出连续的内存空间。

当函数执行到 return 语句时，它会将其要返回的各个值**复制到这些预留好的栈空间**中。函数执行完毕后，控制权返回给调用方。此时，调用方可以直接从它自己的栈帧上（即之前为返回值预留的空间）获取这些返回的值。

现在无论是单返回值还是多返回值都会**先尝试放在寄存器**中，只有寄存器放不下了才会用到栈。

### Go语言普通指针和unsafe.Pointer有什么区别?

普通指针比如`*int`、`*string`，它们有明确的类型信息，编译器会进行类型检查和垃圾回收跟踪。**不同类型的指针之间不能直接转换**，这是Go类型安全的体现。

而unsafe.Pointer是Go的通用指针类型，可以理解为C语言中的 void*，它绕过了Go的类型系统。 unsafe.Pointer可以**与任意类型的指针相互转换**，也可以**与uintptr进行转换来做指针运算**。

另外，普通指针受GC管理和类型约束，unsafe.Pointer不受类型约束但仍受GC跟踪

### unsafe.Pointer与uintptr有什么区别和联系

unsafe.Pointer和uintptr可以相互转换，这是Go提供的唯一合法的指针运算方式。典型用法是先将unsafe.Pointer转为uintptr做算术运算,然后再转回unsafe.Pointer使用。

最关键的**区别在于GC跟踪**。unsafe.Pointer会被垃圾回收器跟踪，它指向的内存不会被错误回收；而uintptr只是一个普通整数，GC完全不知道它指向什么，如果没有其他引用，**对应内存可能随时被回收**。

```go
type slice struct {

    len   int            // 切片的长度（当前存储的元素个数）
```

扩容机制（以 Go 1.18+ 为准）
当你使用 append 且容量不足时，Go 会触发扩容逻辑：

- 小容量：如果期望容量小于 256，则**直接翻倍**（2x）。
- 大容量：如果期望容量超过 256，则不再简单翻倍，而是按照公式 newcap += (newcap + 3*256) / 4 **逐渐增长**。
- 内存对齐：计算完大致容量后，Go 还会根据内存分配器的规格进行向上取整（内存对齐），所以最终分配的 cap 往往比计算出的稍微大一点。

### map实现

​     Map 使用了**哈希表实现，采用链地址法**（通过 Bucket 桶）来解决哈希冲突。

```go
type hmap struct {
    count     int    // 元素个数
    B         uint8  // 桶的数量，数量为 2^B
    buckets   unsafe.Pointer // 指向桶数组的指针
    oldbuckets unsafe.Pointer // 扩容时指向旧桶的指针（用于渐进式搬迁）
    extra     *mapextra      // 溢出桶相关
}
```

`hmap` 中的 `buckets` 成员指向一个连续的内存区域，里面存放的是一个个 `bmap` 结构。每个 `bmap`（俗称 Bucket）最多存放 **8 个键值对**。

桶内部的tophash数组存储桶内**每个key对应哈希值的高8位**，桶内部的数据排列方式是`tophash | K1, K2...K8 | V1, V2...V8`

**插入一个值的全过程**

假设我们有一个 `m := make(map[string]int, 8)`，此时 $B=3$（即 $2^3=8$ 个桶）。我们要插入 `m["Gopher"] = 100`。

1. 计算哈希

        Go 会计算 `"Gopher"` 的 64 位哈希值。   

        假设哈希值（二进制）为：`10010110 ... 00101101`

2. 确定桶位置

        使用哈希值的**低 B 位**（即低 3 位）来决定存放在哪个桶。

        低 3 位是 `101`（十进制 5）。

        因此，这个键值对会被分配到 **第 5 号桶**。

3. 桶内查找与冲突处理

    **检查 Tophash：** 取哈希值的**高 8 位**（假设是 `10010110`）。

    **遍历桶：** 在 5 号桶的 `tophash` 数组里找有没有空位。

    **插入：**

        如果有空位（比如第 1 个 slot 是空的），就把高 8 位填入 `tophash[0]`。

        把 `"Gopher"` 放入 `keys[0]`，把 `100` 放入 `values[0]`。

    **处理冲突（满员情况）：**

        如果 5 号桶的 8 个 slot 全满了，Go 会从 `extra.nextOverflow` 拿一个预分配好的**溢出桶**，挂在 5 号桶后面，然后把数据存入溢出桶。

**查找一个值的全过程**

`m := make(map[string]int, 8)`，此时 $B=3$（即 8 个桶），我们要查找 `m["Gopher"]`。

1. 哈希计算

        Go 首先对键 `"Gopher"` 进行哈希运算，得到一个 64 位的哈希值。

        **示例哈希值（二进制）：** `10010110 ... 00101101`

        这个哈希值会被拆成两部分使用：**高 8 位**用于桶内匹配，**低 B 位**用于定位桶。

2. 定位桶

        取出哈希值的 **低 3 位**（因为 $B=3$）。

        低 3 位是 `101`，即十进制的 **5**。

        Go 运行时直接跳到 `hmap.buckets` 指针指向的内存地址，偏移 5 个 `bmap` 的长度，找到 **5 号桶**。

3. 桶内初筛

        现在进入 5 号桶。Go 不会立刻去对比字符串 `"Gopher"`，而是取出哈希值的 **高 8 位**：`10010110`。

        Go 遍历 5 号桶里的 `tophash[8]` 数组。

        **比对：** 比较 `10010110` 是否等于 `tophash[0]`、`tophash[1]` ... 依此类推。

        **结果：** 假设在 `tophash[2]` 处发现了匹配。

4. 精确核对

        虽然 `tophash[2]` 匹配了，但哈希高 8 位相同并不代表 Key 一定相同（存在哈希碰撞）。

        Go 根据索引 `2`，定位到该桶内 `keys` 区域的第 3 个位置。

        取出内存中的那个 Key，与目标 `"Gopher"` 进行全量比较。

        **如果相同：** 恭喜，根据索引 `2` 到 `values` 区域取出对应的值（如 `100`），查找结束。

        **如果不同：** 继续往后比对剩余的 `tophash` 位（tophash数组中可能存在相同的值）。

5. 处理溢出桶

        如果在当前 5 号桶的 8 个 slot 里都没找到：

            检查该桶的 `overflow` 指针。

            如果指针不为空，说明还有“挂”在外面的溢出桶。

            **跳转：** 进入溢出桶，重复第三步和第四步。

            如果找完了所有溢出桶都没找到，则返回该类型的零值（如 `0`）。

扩容逻辑
Map 的扩容是渐进式的（每次进行写或删操作时搬迁 1-2 个桶），避免瞬间性能抖动。

- 翻倍扩容：当装载因子（元素数量除以桶数量）超过 6.5 时（即元素太多），桶的数量翻倍。
- 等量扩容：当溢出桶过多时，会进行整理，桶的数量不变。
  - 当 B < 15 时，溢出桶数量 >= 2^B；当 B >= 15 时，溢出桶数量 >= 2^15。

### Map遍历为什么是无序的？

Go 的 `map` 底层是哈希表。造成其遍历无序有以下几个技术原因：

- **哈希分布（Hashing）**： 当你向 `map` 插入键值对时，Key 会经过哈希函数计算得到一个哈希值，并根据这个值的**低B位**决定放入哪个 **Bucket（桶）**。由于哈希函数的离散性，Key 在桶中的分布本身就是乱序的，不取决于插入顺序。
- **扩容重排（Rehash/Evacuation）**： 当 `map` 触发扩容时，所有的键值对会从旧桶迁移到新桶。在这个过程中，同一个桶里的数据可能会分流到不同的新桶中，导致原本在物理内存中相邻的数据变得不再相邻。
- **起始位置随机化**：被故意设计在遍历开始时**随机选择一个桶**作为起点，并**随机选择** Bucket 内的**一个 Cell** 开始遍历。

### 如何对Map有序遍历？

最规范的做法是将 Map 的键（Key）取出来放入一个切片中，用 `sort` 包对切片进行排序，然后根据这个有序的切片去遍历 Map。

### Map的Key一定要是可比较的吗？为什么？

Map的Key必须要可比较。

首先，Map会对我们提供的Key进行哈希运算，得到一个哈希值。这个哈希值决定了这个键值对大概存储在哪个位置（也就是哪个“桶”里）。然而，**不同的Key可能会产生相同的哈希**值，这就是“哈希冲突“。当多个Key被定位到同一个”桶“里时，这意味着tophash数组中有多个相同的值，此时，**必须用传入的key与相同tophash对应的key进行比较**，这样才能确保我们操作的是正确的键值对。

### Go语言Map的扩容过程是怎样的？

Go的扩容是渐进式的。它不会在触发扩容时“stop the world”来一次性把所有数据搬迁到新空间，而是**只分配新空间**，然后在后续的每一次**插入、修改或删除**操作时，才会顺便搬迁一两个旧桶的数据。这种设计将庞大的扩容成本分摊到了多次操作中，极大地减少了服务的瞬间延迟（STW），保证了性能的平滑性。

如果是触发双倍扩容，会新建一个buckets数组，新的buckets数量大小是原来的2倍，然后旧buckets数据搬迁到新的buckets。如果是等量扩容，buckets数量维持不变，重新做一遍类似双倍扩容的搬迁动作，把松散的键值对重新排列一次，使得同一个 bucket 中的key排列地更紧密，这样节省空间，存取效率更高

### 可以对Map的元素取地址吗？

无法对 map 的 key 或 value 进行取址。会发生编译报错，这样设计主要是因为map**一旦发生扩容，key和 value 的位置就会改变**，之前保存的地址也就失效了。

### Map 中删除一个key，它的内存会释放么？

**不会**。删除 key 只是把对应的 slot 标记为空（tophash 置为 empty），但bucket数组的空间不会被释放，该位置可以被新值顶替，只有重建 Map （或进行等量扩容）才能真正释放。

### 不要通过共享内存来通信，而要通过通信来共享内存

传统方式：通过共享内存来通信 

- 做法：两个线程通过读写同一个变量（如全局变量、Map）来交换信息。
- 代价：为了防止竞争，必须频繁加锁。
- 痛点：锁竞争会导致性能下降，且容易产生死锁或竞态条件。

Go 的方式：通过通信来共享内存 

- 做法：通过 Channel 发送数据。当一个 Goroutine 将数据的指针发给另一个协程后，它就不再操作这个指针了（自行约定）。
- 核心逻辑：数据的**所有权发生了转移。**
- 优势：
  - 天然同步：Channel 内部保证了 Happens-Before。
  - 降低心智负担：不需要在业务逻辑里到处考虑 Lock/Unlock。
  - 解耦：生产者和消费者不需要知道对方的存在，只需要关注 Channel。

### 缓冲、无缓冲 chan的区别

1. 无缓冲 Channel (make(chan T))
   无缓冲通道也被称为同步通道。
   特性：它的容量为 0，不能存储任何数据。
   行为：
   
       发送者会阻塞，直到有接收者准备好取走数据。
       接收者会阻塞，直到有发送者准备好发送数据。
   
   同步性：它保证了发送和接收是**同时发生**的。这种机制确保了两个 Goroutine 在数据交换的那一刻处于同步状态。

2. 有缓冲 Channel (`make(chan T, capacity)`)
   有缓冲通道类似于一个固定大小的队列。
   特性：拥有一个内部队列，可以暂存 capacity 个元素。
   行为：
   
   发送者：只要缓冲区没满，发送操作就是非阻塞的，数据存入缓冲区后立即返回；只有当缓冲区满时，发送者才会阻塞。
   接收者：只要缓冲区有数据，接收操作就是非阻塞的；只有当缓冲区空时，接收者才会阻塞。

​    同步性：它是异步的。发送者和接收者之间存在解耦。 

### Channel的底层实现原理是怎样的？

 Channel的底层是一个名为 hchan 的结构体，核心包含几个关键组件：

```go
type hchan struct {
    qcount   uint           // 当前缓冲区中的元素个数
    dataqsiz uint           // 环形队列的长度（make(chan T, cap) 中的 cap）
    buf      unsafe.Pointer // 指向环形队列的指针（仅针对缓冲型 channel）
    elemsize uint16         // 元素的大小
    closed   uint32         // 标识 channel 是否关闭
    elemtype *_type         // 元素的类型
    sendx    uint           // 发送队列的下标（记录下次写在 buf 的哪个位置）
    recvx    uint           // 接收队列的下标（记录下次从 buf 的哪个位置读）
    recvq    waitq          // 等待接收的协程队列（Sudog 链表）
    sendq    waitq          // 等待发送的协程队列（Sudog 链表）
    lock     mutex          // 互斥锁，保护 hchan 的所有字段
}
```

1. 环形缓冲区：有缓冲channel内部维护一个固定大小的环形队列，用 buf 指针指向缓冲区，sendx 和recvx 分别记录发送和接收的位置索引。
2. 两个等待队列 sendq和recvq：用来管理阻塞的goroutine。 sendq 存储**因channel满而阻塞的发送者**，recvq 存储**因channel空而阻塞的接收者**。这些队列用双向链表实现，当条件满足时会唤醒对应的goroutine.
3. 互斥锁：hchan 内部有个mutex，所有的发送、接收操作（修改hchan内的字段）都需要先获取锁，用来保证并发安全。

### 向channel发送数据的过程是怎样的？

1. **检查是否关闭**：如果**已关闭，Panic**。
2. **检查接收队列**：如果`recvq`队列不为空，说明有goroutine在等待接收数据，这时会**直接把数据传递给等待的接收者**，跳过缓冲区，这是最高效的路径。同时会**唤醒对应的goroutine继续执行**。
3. **如果没有等待接受者，检查缓冲区**：如果缓冲区未满，将数据写入缓冲区。
4. **如果缓冲区满了需要阻塞等待**：如果缓冲区已满，将当前 G 加入发送队列 (`sendq`)，挂起等待。

**被唤醒后继续执行**。当有接收者从channel读取数据后，会从`sendq`中唤醒一个等待的发送者，被唤醒的goroutine会完成数据发送并继续执行。

### 从Channel读取数据的过程是怎样的

1. **首先检查是否有等待的发送者**。如果`sendq`队列不为空，说明有goroutine在等待发送数据。对于**无缓冲channel，会直接从发送者那里接收数据**；对于**有缓冲channel，会先从缓冲区取数据，然后把等待发送者的数据放入缓冲区**，这样保持FIFO顺序。
2. **如果没有等待发送者，尝试从缓冲区读取**。检查`qcount > 0`，如果缓冲区有数据，就从`buf[recvx]`位置取出数据，然后更新`recvx`索引和`qcount`计数。
3. **缓冲区为空时需要阻塞等待**。创建`sudog`结构体包装当前goroutine，加入到`recvq`等待队列，调用`gopark`进入阻塞状态。当有发送者写入数据时会被唤醒继续执行。

**从已关闭channel读取有特殊处理**。如果channel已关闭且缓冲区为空，会返回**零值和false标志**；如果缓冲区还有数据，可以正常读取直到清空。这就是为什么`v, ok := <-ch`中的ok能判断channel状态的原因。

### 从一个已关闭Channel仍能读出数据吗？

从一个有缓冲的 channel 里读数据，当 channel 被关闭，依然能读出有效值。只有当返回的 ok 为 false 时，读出的数据才是无效的。

### Channel在什么情况下会引起内存泄漏？

Channel引起内存泄漏最常见的是引起goroutine泄漏从而导致的间接内存泄漏，当**goroutine阻塞在channel操作上永远无法退出**时，**goroutine本身和它引用的所有变量都无法被GC回收**。比如一个goroutine在等待接收数据，但发送者已经退出了，这个接收者就会永远阻塞下去。或者**select语句使用不当**，在没有default分支的select中，如果所有case都无法执行，goroutine会永远阻塞。出现内存泄漏

### Channel会产生panic的场景

* 向已经关闭的 Channel 发送数据 （**向nil channel发送数据、读取数据会永久阻塞**）
* 重复关闭同一个 Channel
* 关闭一个 nil Channel（没有使用make进行初始化就是nil）

### 什么是select？

select是Go语言专门为channel操作设计的**多路复用控制结构**，类似于网络编程中的select系统调用。

核心作用是**同时监听多个channel操作**。当有多个channel都可能有数据收发时，select能够选择其中一个可执行的case进行操作，而不是按顺序逐个尝试。比如同时监听数据输入、超时信号、取消信号等。

### select的执行机制是怎样的？

select的执行机制是随机选择。如果多个case同时满足条件，Go会**随机选择一个执行**，这避免了饥饿问题。如果没有case能执行就会执行default，**如果没有default，当前goroutine会阻塞等待。**

### select的实现原理是怎样的？

定义了一个数据结构**scase**表示每个`case`语句(包含`default`)。scase结构包含channel指针、操作类型等信息。

1. 随机化

为了保证公平性，`select` 在执行前会通过一个随机算法**打乱所有 case 的轮询顺序**。防止某些 case 因为排在前面而被频繁触发（饥饿问题）。

2. 轮询

按照随机生成的顺序，依次检查每个 `case`：

- 如果是 `send` 操作且 Channel 准备好接收，或者 `recv` 操作且 Channel 有数据，则立即执行该 `case` 并跳转到对应的代码块。

- 如果存在 `default` 分支且所有 Channel 都没准备好，则直接执行 `default`。

- `select` 只要执行了其中任何一个 `case`（包括 `default`），它就会立即结束整个 `select` 块的逻辑，继续执行 `select` 后面的代码。
3. 挂起

如果没有任何 case 准备好，且没有 `default` 分支，**当前的 Goroutine 就会被阻塞**：

调度器会将当前的 Goroutine（G）包装成**多个 `sudog`**。将这些 `sudog` 分别放入 **`select` 中涉及到的所有 Channel 的等待队列**（发送队列或接收队列）中。调用 `gopark` 挂起本协程，等待被唤醒。

4. 唤醒

一旦其中任何一个 Channel 准备好了，它会唤醒该协程：协程醒来后，会先将自己从其他所有 Channel 的等待队列中移除（通过 **锁** 保证原子性）。执行对应的 case 逻辑。

### 除了 mutex 以外还有那些方式安全读写共享变量？

除了Mutex，主要还有**信号量**、**通道（Channel）**，**原子操作(atomic)** 这几种方式。

- 信号量的实现其实跟mutex差不多，实现起来也很方便，主要通过信号量计数来保证。

- channel是Go最推崇的方式，它通过通信来传递数据所有权，从根源上避免竞争，更适合复杂的业务逻辑；

- 原子操作则针对最简单的**整型或指针**等进行无锁操作，**性能最高**，常用于实现计数器或状态位。

### Go 语言是如何实现原子操作的？

Go语言实现原子操作，其根本是**依赖底层CPU硬件提供的原子指令**，而不是通过操作系统或更上层的锁机制。

### 聊聊原子操作和锁的区别？

原子操作和锁最核心的区别在于它们的**实现层级**和**保护范围**。

**原子操作**是CPU硬件层面的，它保证**对单个数据**（通常是整型或指针）的单次读改写操作是绝对不可分割的，性能极高，因为它不涉及操作系统内核的介入和goroutine的挂起。

**锁**则是操作系统或语言运行时提供的，它保护的是一个**代码块**（临界区），而不仅仅是单个变量。当**获取锁失败时，它会让goroutine休眠，而不是空耗CPU**。虽然锁的开销远大于原子操作，但它能保护一段复杂的、涉及多个变量的业务逻辑。

所以，对于简单的计数器或标志位更新，用原子操作追求极致性能；而只要需要保护一段逻辑或多个变量的一致性，就必须用锁。

### Mutex 有几种模式

Go的`Mutex`主要有两种模式：**正常模式（Normal Mode）和饥饿模式（Starvation Mode）**。

1. **正常模式**：这是默认模式，讲究的是性能。新请求锁的goroutine会和等待队列头部的goroutine竞争，新来的goroutine有几次"**自旋**"的机会，如果在此期间锁被释放，它可以大概率抢到锁。这种方式吞吐量高，但可能会导致队列头部的goroutine等待很久，即"不公平"。
2. **饥饿模式**：当一个goroutine**在等待队列中等待超过1ms后**，Mutex就会切换到此模式，讲究的是公平。在此模式下，锁的所有权会**直接从解锁的goroutine移交给等待队列的头部**，新来的goroutine不会自旋，**必须排到队尾**。这样可以确保队列中的等待者不会被"饿死"。

当等待队列为空，或者一个goroutine拿到锁时**发现它的等待时间小于1ms**，饥饿模式就会结束，切换回正常模式。这两种模式的动态切换，是Go在性能和公平性之间做的精妙平衡。

### sync.Once 的作用是什么，讲讲它的底层实现原理？

`sync.Once`的作用是**确保一个函数在程序生命周期内，无论在多少个goroutine中被调用，都只会被执行一次**。它常用于单例对象的初始化或一些只需要执行一次的全局配置加载

`sync.Once`保证代码段只执行1次的原理主要是其内部维护了一个标识位，当它 == 0 时表示还没执行过函数，此时会加锁，再检查一次标识位（避免在竞争锁的时候已经被其他人执行了），确定无误后执行对应函数，再修改标识位（原子操作）。后续再执行时发现标识位 != 0，则不会再执行后续动作了，通过锁和原子操作双重锁定。

### WaitGroup 是怎样实现协程等待？

`WaitGroup`实现等待，本质上是**一个原子计数器和一个信号量的协作**。

调用`Add`会增加计数值，`Done`会减计数值。而`Wait`方法会检查这个计数器，如果不为零，当前goroutine通过信号量就被挂起休眠。直到最后一个`Done`调用将计数器清零，它就会通过这个信号量，一次性唤醒所有在`Wait`处等待的goroutine，从而实现协程等待。

实际上原子计数器是 `state atomic.Uint64 // 高 32 位是计数器，低 32 位是等待者数量`当进行Add或Done时增加或减少高32位，当wait时检查高32位是否大于0，大于0则增加低32位并进入休眠，Done执行完导致高32位为0时，检查低32位，如果大于0，则唤醒所有等待者。

### sync.Map的底层原理

 通过“**读写分离**”和“**空间换时间**”来减少锁的竞争。适用于**读多写少** 或者**多个协程写入不同 Key** 的场景

```go
type Map struct {
    mu Mutex                        // 互斥锁，保护 dirty 层的读写和 read 到 dirty 的拷贝
    read atomic.Value               // 存储 readOnly 结构体，支持无锁并发读
    dirty map[any]*entry            // 可写层，包含最新写入的数据。操作必须加锁
    misses int                      // 记录在 read 中未命中，去 dirty 中查找的次数
}

type readOnly struct {
    m       map[any]*entry          // 实际存储数据的字典
    amended bool                    // 标志位：dirty 中是否包含 m 中没有的新 Key,是则为true
}

type entry struct {
    p unsafe.Pointer                // 指向实际存储 Value 的指针
}
```

- **`read`** ：负责处理绝大多数（少数可能只存在于dirty中）的高频**读取和更新**操作。它被设计为完全**无锁**的。read中的key一旦生成就不能修改了。你不能往里面新增 Key（只能往dirty新增），也不能从里面把 Key 物理删除（只能标记）。
- **`dirty`** ：负责接纳所有**全新**的 Key，操作时必须加锁，满足条件后，会被提升替换掉read。
- **`entry`**：`read` 和 `dirty` 中相同的 Key 指向的是内存中**同一个 `entry` 实例**。这意味着，只要修改了 `entry.p`，`read` 和 `dirty` 都能立刻看到变化

`entry.p` 指针有三种状态变化。

1. **正常状态 (`*value`)**：指针指向实际的数据所在的内存地址。

2. **软删除状态 (`nil`)**：数据被删除了，但这个 Key 依然同时存在于 `read` 和 `dirty` 中。

3. **彻底抹除状态 (`expunged`)**：数据被删除了，且这个 Key **只存在于 `read` 中，在 `dirty` 中不存在该key**（从read向dirty中拷贝数据的时候）。

为什么需要 `expunged`？

当程序需要写入一个全新的 Key 时，如果此时 `dirty` 为空（dirty刚刚被提升为read），Go 需要把 `read` 中的数据拷贝一份初始化给 `dirty`。
在拷贝时，Go **不会**拷贝那些指针为 `nil`（已删除）的键值对，以节省内存。并将它们标记为expunged。
如果在 `read` 中看到 `expunged`，说明这个 Key 绝对不存在于`dirty`中。如果对这个key进行更新操作，那么必须对dirty加锁插入到其中。（如果是nil，说明该key存在于dirty中，可以直接通过read的原子操作对这个entry更新，不需要加锁）

有了expunged，可以不用将read中为nil的数据拷贝到dirty中（防止dirty无效臃肿），可以通过expunged来判断key是否存在于dirty中，避免数据丢失。同样根据expunged来判断是否要加锁才能对key更新。

1. 读取逻辑 (Load)
   
   1. 首先在无锁的 `read` 中找。如果找到了，且状态不是 `nil` 或 `expunged`，直接返回数据。
   2. 如果 `read` 里没有，且系统标记 `dirty` 里有新数据（amended字段），就会加锁去 `dirty` 里找。
   3. 只要去 `dirty` 里找了一次，就会给 `misses` 计数器加 1。
   4. 如果 `misses` 的次数达到了 `dirty` 元素的总数，说明“穿透”太频繁了。系统会直接把当前的 `dirty` 升级为新的 `read`，然后把后台的 `dirty` 清空（设为 `nil`）

2. 写入逻辑 (Store) - 写入 `m.Store("user_2", "Alice")`
   
   1. 如果 `"user_2"` 已经在 `read` 中，且状态不是 `expunged`（是nil也可以直接更新），直接原子替换 `entry` 里的指针为 "Alice"。操作结束
   
   2. 如果 `read` 里没有，或者状态是 `expunged`，必须加锁操作。
      
      1. Key 在 `read` 中是 `expunged` 状态。
         
         - 系统会把状态从 `expunged` 解封为 `nil`，然后把它**重新强行塞回** `dirty` 中。
         
         - 最后更新 `entry` 的值为 "Alice"。
      
      2. 是一个完全陌生的新 Key，必须放进dirty中
         
         - 如果此时 `dirty` 是空的，系统会遍历 `read`，把里面所有存活的 Key 浅拷贝到 `dirty` 里。同时，把 `read` 里那些状态为 `nil` 的废弃 Key，全部打上 `expunged` 标记，不进行拷贝。
         
         - 把新 Key 放入 `dirty`。(如果dirty中存在该key，直接修改即可)

3. 删除逻辑 (Delete) - `m.Delete("user_3")`
   
   1. 如果 `"user_3"` 在 `read` 中存在，系统不会去动字典结构，而是直接通过原子操作把 `entry` 的状态改为 `nil`。
   2. 如果 `"user_3"` 只存在于刚写入的 `dirty` 中（还没来得及晋升到 `read`），那就加锁，直接用 Go 原生的 `delete()` 函数把它从 `dirty` 字典里彻底抹掉。

### Go语言的Context

go语言里的context实际上**是一个接口**，提供了`Deadline()，Done()，Err()`以及`Value()`四种方法。它在Go 1.7 标准库被引入。

它本质上是一个**信号传递和范围控制的工具**。它的核心作用是在一个请求处理链路中（跨越多个函数和goroutine），优雅地传递**取消信号（cancel）、超时（timeout）和截止日期（deadline）**，并能携带一些**键值对数据**。

这个接口定义了四个核心方法，它们共同构成了一套关于**截止时间、取消信号和请求范围值**的协定：

- `Deadline()` - 返回一个时间点，告知任务何时应该被取消。
- `Done()` - 返回一个channel，当`Context`**被取消或超时**，这个channel会被关闭。这是**goroutine监听取消信号**的核心。
- `Err()` - 在`Done()`的channel关闭后，它会解释关闭的原因，是主动取消（`Canceled`）还是超时（`DeadlineExceeded`）。
- `Value()` - 允许`Context`在调用链中携带请求范围的键值对数据。

### context的应用场景

​    context包是管理并发控制、生命周期和跨层级传递数据的标准方式。

1. 超时控制 (Timeout)
   在访问数据库、调用第三方 API 或执行耗时操作时，我们**不能让请求无限期等待**。
   
   ​    用法：context.WithTimeout

2. 取消信号传递 (Cancel)
   当一个**父级请求**因为某种原因（如用户关闭了浏览器）被取消时，该**请求派生出的所有下游 Goroutine 都应该立即停止执行**，以节省资源。
   
   ​    场景：主请求取消后，停止相关的数据库查询和缓存写入。
   
   ​    用法：context.WithCancel

3. 截止时间控制 (Deadline)
   与超时类似，但它的是一个具体的时间点（Time object），而不是时间段（Duration）。
   用法：context.WithDeadline

4. 跨层级元数据传递 (Value)
   在请求的生命周期内，在不同的函数、中间件或服务层之间传递一些公共信息。
   场景：传递 TraceID（链路追踪）、UserID（当前登录用户）、Token。
   用法：context.WithValue
   注意：不要用它传递业务参数。它应该是与业务逻辑无关、用于横向关注点（如日志、监控）的数据。

注意事项：

1. 不要将 Context 存储在结构体中：Context 应该作为函数的第一个参数显式传递，变量名通常叫 ctx。
2. 不要传递 nil Context：如果你不确定要传什么，使用 context.TODO()，而不是 nil。
3. Context 是不可变的：每次衍生（WithCancel, WithValue 等）都会返回一个新的子 Context。当**父级被取消时，所有子级都会收到信号**。
4. 及时调用 Cancel（除了WithValue之外的创建的方式会返回Cancel函数）：无论是超时还是正常结束，只要使用了派生的 Context，一定要**记得 defer cancel()**，否则会导致 Goroutine 泄露或计时器无法释放。

### Context如何被取消

Context的取消是通过**channel关闭信号**实现的，主要有三种取消方式。

首先是**主动取消**，通过`context.WithCancel`创建的Context会返回一个cancel函数，调用这个函数就会关闭内部的done channel，所有监听这个Context的goroutine都能通过`ctx.Done()`收到取消信号。

其次是**超时取消**，`context.WithTimeout`和`context.WithDeadline`会启动一个定时器，到达指定时间后自动调用cancel函数触发取消。

最后是**级联取消**，当父Context被取消时，所有子Context会自动被取消，这是通过Context树的结构实现的。

### 接口和泛型

1. 接口 (Interface)：解决的是“行为抽象”问题，多个类型有共同的操作
   接口定义了一组行为（方法）。它不关心对象是什么，只关心对象能做什么。

2. 泛型 (Generics)：解决的是“类型安全与代码复用”问题，逻辑相同但类型不同
   泛型允许你在编写函数或数据结构时使用类型占位符，直到使用时才确定具体类型。

3. interface{} 和 any 的关系
   any是别名

4. 使用泛型相比接口的优势
   虽然泛型和接口在某些场景下可以互换，但泛型有以下显著优势：
   
   ​    1. 类型安全（无需断言），从运行时提前到编译时
   
   ​        接口：如果你把数据存入 []interface{}，**取出时**必须进行**类型断言** v.(int)，这在运行时**可能触发 panic**。
   ​        泛型：在编译阶段就确定了类型。
   
   ​    2. 消除运行时开销
   
   ​        接口：使用接口必须要进行运行时的类型断言。
   ​        泛型：使用泛型会在**编译时生成特定类型的版本**（或类似机制）。它更接近于静态调用，对编译器优化更友好，减少了运行时的装箱/拆箱开销。
   
   ​    3. 泛型表达力更强
   
   ​    例子：写一个 Max(a, b T) T 函数。
   
      ​    用接口很难实现，因为你需要定义一个 Comparable 接口，让不同的类型实现这个接口，而且返回值往往需要类型断言回具体类型。
      ​    用泛型非常简单：`func Max[T constraints.Ordered](a, b T) T { ... }`。它能保证输入和输出的类型是完全一致的。

### interface的底层原理是怎样的？

Go的interface底层有两种数据结构：**eface和iface**。

**eface是空interface{}的实现**，只包含两个指针：`_type`指向类型信息，`data`指向实际数据。这就是为什么空接口能存储任意类型值的原因，通过类型指针来标识具体类型，通过数据指针来访问实际值。

**iface是带方法的interface实现**，包含`itab`和`data`两部分。`itab`是核心，它存储了接口类型、具体类型，以及方法表。方法表是个函数指针数组，保存了该类型实现的**所有接口方法的地址。**

### 类型转换和断言的区别是什么

类型转换、类型断言本质**都是把一个类型转换成另外一个类型**。不同之处在于，类型断言是对接口变量进行的操作。对于**类型转换**而言，类型转换是**在编译期确定的强制转换**，转换前后的两个类型要相互兼容才行，语法是T(value)。而**类型断言**是运行期的动态检查，专门用于从接口类型中提取具体类型，语法是value.(T)

- **安全性差别很大**：类型转换在编译期保证安全性，而类型断言可能在运行时失败。所以实际开发中更常用安全版本的类型断言`value, ok := x.(string)`，通过ok判断是否成功。

- **使用场景不同**：类型转换主要解决数值类型、字符串、切片等之间的转换问题；类型断言主要用于接口编程，当你拿到一个interface{}需要还原成具体类型时使用。

- **底层实现也不同**：类型转换通常是简单的内存重新解释或者数据格式调整；类型断言需要检查接口的底层类型信息，涉及到runtime的类型系统。

### 什么是反射？

反射是指计算机程序在运行时可以访问、检测和修改它**本身状态或行为**的一种能力。

### Go语言如何实现反射？

Go语言反射是通过接口来实现的，一个接口变量包含两个指针结构：一个指针指向**类型信息**，另一个指针指向**实际的数据**。当我们将一个具体类型的变量赋值给一个接口时，Go就会把这个变量的类型信息和数据地址都存到这个接口变量里。

有了这个前提，Go语言就可以通过`reflect`包的`Type`和`ValueOf`这两个函数（会将普通变量转为接口变量）读取接口变量里的类型信息和数据信息。把这些内部信息"解包"成可供我们检查和操作的对象，完成在运行时对程序本身的动态访问和修改。

### 如何比较两个对象完全相同

**最直接的是用reflect.DeepEqual**，这是标准库提供的深度比较方法，能**递归比较**结构体、切片、map等复合类型的所有字段和元素。比如`reflect.DeepEqual(obj1, obj2)`，它会**逐层比较内部所有数据**，包括指针指向的值。

**对于简单类型可以直接用==操作符**，但这只适合基本类型、**数组**、结构体等可比较类型。需要注意slice、map、function这些类型是不能直接用==比较的，会编译报错。

**实际项目中更推荐自定义Equal方法**，根据业务需求定义相等的标准。比如对于用户对象，可能只需要比较ID和关键字段，而不需要比较时间戳这种辅助字段。这样既提高了性能，又符合业务语义。

### GMP 调度模型

它负责将成千上万个 Goroutine (G) 映射到有限的操作系统线程 (M) 上执行。
其核心组件如下：

- G (Goroutine)： 即协程。它**保存了任务执行的上下文**（栈、程序计数器 PC 等）。G 只是一个管理任务的数据结构，不直接执行。
- M (Machine)： 操作系统线程。由内核调度，是真正的执行计算资源。**M 想要执行 G，必须先绑定一个 P**。
- P (Processor)： 逻辑处理器（上下文）。P 维护了一个**本地局部队列**，里面存放着待运行的 G。
  - P 的数量决定了系统并行的上限（通常等于 CPU 核心数，可通过 GOMAXPROCS 设置）。
  - 核心作用：**解耦 G 和 M。即使某个 M 因为系统调用阻塞了，P 可以带着剩下的 G 去找别的 M 执行。** P中保存本地队列，避免严重的锁竞争。

核心调度机制：

- Work Stealing：如果某个 P (比如 P1) 的本地队列空了，而它绑定的 M 也没活干了，M 不会直接休眠。它会去全局队列找（需要加锁），如果全局队列也没有，它就会**随机挑选另一个 P (比如 P2)，从 P2 的本地队列里“偷走”一半的 G** 放到自己的队列里执行。
- Hand Off：当一个 G 执行了阻塞的**系统调用**（比如读写磁盘文件 CGO 操作），底层执行该调用的线程 M 会被操作系统阻塞进入内核态。
  此时，**绑定的 P 会立刻果断地“抛弃”这个被阻塞的 M**，带着自己本地队列里剩下的 G，去寻找另一个空闲的 M（或者新建一个 M）继续执行
- 抢占式调度：如果一个 G 里面写了一个没有 IO 阻塞的死循环，它会不会永远霸占当前的 M 和 P？
  在 Go 1.14 之前会，但现在不会了。Go Runtime 在后台运行了一个特殊的系统监控线程叫做 **`sysmon`**（它不需要绑定 P）。 `sysmon` 会周期性扫描所有的 P，如果发现某个 G 连续运行超过 **10 毫秒**，它就会向该 M 发送一个底层的系统信号（如 `SIGURG`），强行中断它的执行，把这个 G 踢回全局队列，让其他 G 有机会运行。

### GMP能不能去掉P层？

GMP中的P层理论上可以去掉，但会带来严重的性能问题。

**去掉P的后果**：如果直接变成GM模型，所有M都需要从**全局队列**中获取goroutine，这就需要**全局锁保护**。在高并发场景下，大量M争抢同一把锁会造成严重的**锁竞争**，CPU大部分时间都浪费在等锁上，调度效率急剧下降。

**P层的价值**：P的存在实现了**无锁的本地调度**。每个P维护独立的本地队列，M绑定P后可以直接从本地队列取G执行，大部分情况下都不需要全局锁。只有本地队列空了才去偷取，这大大减少了锁竞争。

### P和M在什么时候会被创建？

**P的创建时机**：P**在调度器初始化时一次性创建**。在`schedinit()`函数中会调用`procresize()`，根据`GOMAXPROCS`值创建对应数量的P对象，存储在全局的`allp`数组中。之后P的数量基本固定，只有在调用`runtime.GOMAXPROCS()`动态调整时才会重新分配P。

**M的创建时机**：M采用**按需创建**策略。**初始只有m0存在**，当出现以下情况时会创建新的M：

- 所有现有M都在执行阻塞的系统调用，但还有可运行的goroutine需要执行
- 通过`startm()`函数发现没有空闲M可以绑定P执行goroutine
- M的数量受`GOMAXTHREADS`限制，默认10000个

**创建流程**：新M通过`newm()`函数创建，它会调用`newosproc()`创建新的系统线程，并为这个M分配独立的g0。创建完成后，新M会进入`mstart()`开始调度循环。

### m0是什么，有什么用

m0是在Go启动时创建的第一个M，m0对应程序启动时的主系统线程，它在Go程序的整个生命周期中都存在。与其他通过`runtime.newm()`动态创建的M不同，m0是在程序初始化阶段静态分配的，有专门的全局变量存储。

m0主要负责执行Go程序的**启动流程**，包括调度器初始化、内存管理器初始化、垃圾回收器设置等。它会**创建并运行**第一个用户goroutine来**执行`main.main`函数**。在程序运行期间，m0也参与正常的goroutine调度，和其他M没有本质区别。m0在程序退出时还负责处理清理工作，比如等待其他goroutine结束、执行defer函数等。

### g0是一个怎样的协程，有什么用？

g0是一个特殊的goroutine，不是普通的用户协程，而是**调度协程**，**每个M都有自己的g0**。它使用系统线程的原始栈空间，而不是像普通goroutine那样使用可增长的分段栈。g0的栈大小通常是8KB，比普通goroutine的2KB初始栈要大。

**核心作用**：g0专门负责**执行调度逻辑**，包括goroutine的创建、销毁、调度决策等。当M需要进行调度时，会从当前运行的用户goroutine切换到g0上执行`schedule()`函数。g0还负责处理垃圾回收、栈扫描、信号处理等运行时操作。

**运行机制**：正常情况下M在用户goroutine上运行用户代码，当发生调度事件时（如goroutine阻塞、抢占、系统调用返回等），M**会切换到g0执行调度器代码**，选出下一个要运行的goroutine后再切换过去。

**为什么需要g0**：因为调度器代码不能在普通goroutine的栈上执行，那样会有栈空间冲突和递归调度的问题。g0提供了一个独立的执行环境，确保调度器能安全稳定地工作。

### Goroutine被中途挂起切换的情况

这种情况主要分为 “主动挂起” 和 “被动抢占” 两大类。

1. 等待 I/O 操作（最常见）
   当 Goroutine 发起一个网络请求或文件读写时，如果数据还没准备好，它会被挂起。
   
   - 机制：Go 利用 Netpoller（网络轮询器）。G 会被设置为等待状态，并从当前的物理线程（M）上解绑。
   - 结果：调度器让出该 M 去执行其他的 G，直到 I/O 数据就绪，Netpoller 会通知调度器将该 G 重新放入运行队列。

2. 系统调用 (System Call)
   当代码执行了阻塞式的系统调用（如打开文件、获取系统状态等）时：
   
   - 机制：由于系统调用会阻塞内核线程（M），Go 调度器会执行 Hand-off（移交） 策略。它会将该 M 暂时从调度器中剥离，并让出该处理器（P），启动或唤醒一个新的 M 来接管该 P 上的其他 G。
   - 恢复：当系统调用返回后，原 G 会尝试寻找可用的 P 继续运行。

3. 基于信号的抢占式调度 (Preemption)
   如果一个 Goroutine 是纯计算密集型的（比如一个死循环 for {}），它不会主动让出 CPU。为了防止它饿死其他 G，Go（从 1.14 版本开始）引入了基于信号的异步抢占。
   
   - 机制：系统监控线程（sysmon）发现某个 G 运行时间过长（超过 10ms），就会向该 G 所在的线程发送一个 SIGURG 信号。
   - 结果：该 G 收到信号后会被强制挂起，放回全局队列，调度器转而执行其他 G。

4. 等待同步原语 (Channel/Mutex)
   当 Goroutine 尝试进行以下操作但无法立即完成时：
   
   - Channel 阻塞：向满的 Channel 发送数据，或从空的 Channel 接收数据。
   - 锁竞争：请求一个已被其他 G 持有的 sync.Mutex 或 sync.WaitGroup。
   - 机制：该 G 会进入阻塞状态，并被放入该锁或 Channel 的 等待队列（Sudog） 中。调度器立即调度下一个 G。

5. 主动让出 (Runtime.Gosched)
   开发者可以在代码中手动调用 runtime.Gosched()。
   
   - 逻辑：这相当于 G 对调度器说：“我现在的任务不紧急，可以把 CPU 让给别人，但我待会儿还要跑。”
   - 结果：G 会被放回本地运行队列的末尾，调度器重新选择下一个 G。

### Go是怎么进行内存分配的？

Go 的内存管理主要由三个组件组成：**mcache**、**mcentral** 和 **mheap**。

1. mcache（线程本地缓存）
- **归属**：绑定在 GMP 模型中的 **P**（Processor）上。
- **特性**：每个 P 都有一个独立的 `mcache`。由于 P 是单线程运行的，因此从 `mcache` 分配对象**不需要加锁**，性能最高。
- **存储**：缓存了各种大小规格的空闲内存块（mspan）。
3. mcentral（全局中心缓存）
- **归属**：全局，所有P共享，按大小规格（size class）分类管理，这样mcache来申请时，只需要对某个规格内存加锁即可。
- **特性**：当 `mcache` 的内存用完时，会向 `mcentral` 申请。因为 `mcentral` 是全局共享的，所以申请时**需要加锁**。
3. mheap（全局堆内存）
- **归属**：整个 Go 运行时只有一个 `mheap`。
- **特性**：当 `mcentral` 也没有空闲空间时，会向 `mheap` 申请。如果 `mheap` 也没内存了，则会向操作系统（如 `mmap`）申请。
- **存储**：管理的是大块内存（Pages）。

**`mspan`** 是 Go 内存管理的基本单位。

- 它是由一个或多个连续的 **8KB 页面**（Page）组成的双向链表。
- 每一个 `mspan` 会按照固定的规格（共 67 种 size class）切分成大小相等的内存块（Object）。

### 逃逸分析

作用：确定一个变量应该分配在栈上还是堆上。
为什么需要逃逸分析？
    栈（Stack）： 分配和回收极快（仅需移动栈指针），随着函数返回自动释放，压力小。
    堆（Heap）： 需要垃圾回收（GC）来清理。如果所有变量都放堆上，GC 压力会巨大，导致程序变慢。

什么情况会逃逸？
     1. 函数返回局部变量的指针： 如果函数返回一个局部变量的地址，该变量在函数结束后必须继续存在，因此必须分配在堆上
     2. 接口（interface{}）类型存储
     3. 闭包（Closure）引用：闭包函数如果引用了外部函数的局部变量，为了保证闭包在外部函数退出后仍能正常运行，这些变量会逃逸到堆。
     4. 变量空间过大（超过栈大小限制）或由于动态大小：在 `make` 切片时，如果长度是**变量**而不是常量，编译器无法在编译期确定其大小，通常也会逃逸。

### Go语言在什么情况下会发生内存泄漏

**goroutine泄漏**

**slice引用大数组**：当slice引用一个大数组的小部分时，整个底层数组都无法被GC回收。解决方法是使用copy创建新的slice。

**map元素过多**：map中删除元素只是标记删除，底层bucket不会缩减。如果map曾经很大后来元素减少，内存占用仍然很高。

**定时器未停止**：`time.After`或`time.NewTimer`创建的定时器如果不手动停止，会在heap中持续存在直到**触发定时**。

### Goroutine泄露

指启动了一个 Goroutine，但由于某种原因（如逻辑阻塞或死循环），它**永远无法结束**，也无法被垃圾回收（GC）销毁。

场景：

1. Channel 导致的阻塞
   
   发送者阻塞：向一个没有接收者的无缓冲 Channel 发送数据。
   接收者阻塞：从一个永远不会有数据存入、也永远不会关闭的 Channel 接收数据。

2. Select 语句没有 Default 分支，且所有case都不能执行

3. Goroutine 试图获取一个永远不会被释放的锁。

### Go语言发生了内存泄漏如何定位和优化？

**定位工具**：

- **pprof**：最重要的工具，通过`go tool pprof http://localhost:port/debug/pprof/heap`分析堆内存分布，`go tool pprof http://localhost:port/debug/pprof/goroutine`分析goroutine泄漏
- **trace工具**：`go tool trace`可以看到goroutine的生命周期和阻塞情况
- **runtime统计**：通过`runtime.ReadMemStats()`监控内存使用趋势，`runtime.NumGoroutine()`监控协程数量

**定位方法**：我通常先看内存增长曲线，如果内存持续上涨不回收，就用pprof分析哪个函数分配内存最多。如果是goroutine泄漏，会看到goroutine数量异常增长，然后分析这些goroutine阻塞在哪里。

**常见优化手段**：

- **goroutine泄漏**：使用context设置超时，确保goroutine有退出机制，避免无限阻塞
- **channel泄漏**：及时关闭channel，使用select+default避免阻塞
- **slice引用优化**：对大数组的小slice使用copy创建独立副本
- **定时器清理**：手动调用`timer.Stop()`释放资源

### 垃圾回收机制

1. 三色标记法
   为了实现并发扫描（即 GC 和用户程序同时运行），Go 将对象分为三种颜色：
   
   白色：潜在的垃圾，其内存可能会被回收。
   灰色：活跃对象，但其**引用的对象**（其内存储了**内存地址**的字段）尚未被扫描。
   黑色：活跃对象，且其引用的对象已全部扫描完成。

​    GC 过程：
​        起初所有对象都是白色。
​        从根对象（栈、全局变量）开始，将其标记为灰色。
​        遍历灰色对象，将它们引用的对象转为灰色，自身转为黑色。
​        重复此过程，直到没有灰色对象。
​        清理剩下的白色对象。

2. 混合写屏障
   
   这是 Go 1.8 引入的黑科技，它解决了在并发标记期间，用户程序修改指针导致对象丢失的问题。

减少GC停顿时间的优化思路
    本质是**减少堆内存的分配压力**和**减少扫描对象数量**。

### 如何解决并发标记时，用户程序并发修改对象引用问题的？

核心挑战是防止"对象消失"现象：**当黑色对象新增对白色对象的引用，同时灰色到白色的引用被删除时，白色对象可能被错误回收**。Go采用**混合写屏障**策略，在指针赋值时执行额外逻辑：**新建引用时将目标对象着为灰色，删除引用时将被删对象（被指向的对象）标为灰色**（在一条赋值语句中两个都会执行），这样确保对象不会丢失在标记过程中。--- **`丢弃的不放过，新增的要变灰。`**

同时Go维护了**弱三色不变性**：允许黑色对象指向白色对象，但要保证从某个灰色对象出发，存在一条可以到达该白色对象的路径。解决了**栈上不加写屏障也能保证安全**的难题。

当栈上的黑色对象 A（栈操作频繁）想要引用 **白色对象 C**（C对其他灰色对象引用着），出于性能考虑不会采用写屏障保护，由于弱三色不变性，也同样是安全的。当C被其他灰色对象删除引用时，会根据删除写屏障将它设为灰色。

### 什么是写屏障、混合写屏障，如何实现？

写屏障是编译器在指针赋值操作中插入的一段代码，当执行`*slot = ptr`这样的指针赋值时，写屏障会**在赋值前后执行特定逻辑来标记相关对象，防止并发标记过程中对象被错误回收**。

写屏障的核心任务是防止**对象丢失**。对象丢失需同时满足两个条件：

1. **黑指向白**：黑色对象引用了白色对象。

2. **灰断开白**：灰色对象删除了指向该白色对象的唯一引用链。

Dijkstra 插入写屏障 (插入标灰) - 解决第一个条件：

- 当黑色对象引用白色对象时，强制将白色标灰。

- 为了性能，栈上不加屏障。这意味着栈上的黑色对象（已经被扫描过的）可能引用白色对象而不被感知。为了安全，必须在标记结束前 **STW 并重新扫描（Re-scan）整个栈**，这在海量协程场景下是性能灾难。

Yuasa 删除写屏障 (删除标灰) - 解决第二个条件：

- 当删除一个引用时，将旧的指向对象标灰。

- 因为栈上没有写屏障，在标记开始的瞬间，必须通过一次短暂的 STW 记录下所有根对象（栈、全局变量）的状态快照。通过快照保证 **栈上还没扫描到的对象被修改引用** 不会丢失。

混合写屏障的精髓是**利用堆上的屏障逻辑，换取栈上的零屏障**：

当执行 `*slot = ptr` 时，底层触发：

1. **Shade(*slot)**：将原先被引用的对象（旧值）标记为灰色。保证了即使栈上的黑色对象偷偷引用了白色对象，只要这个白色对象之前在堆里（即将被断开引用），它就会被标灰保命。这消灭了 Dijkstra 对 **“栈重扫”** 的依赖。

2. **Shade(ptr)**：将新投入引用的对象（新值）标记为灰色。保证了任何新建立的引用都能被实时捕捉。这消灭了 Yuasa 对 **“起始快照”** 的依赖。

四大准则：

- **栈上零屏障**：栈上赋值 `local_var = ptr` 没有任何额外开销，保证了 Go 处理海量任务的基础性能。

- **栈扫描即黑化**：GC 开始时，瞬间扫描所有 Goroutine 栈，将存活对象全部标记为**黑色**

- **栈上新对象黑化**：标记期间，所有在栈上新生成的对象直接标黑。结合插入标灰消除了Yuasa的缺陷。

- **堆上全屏障**：堆上指针修改，同时执行“删除标灰”和“插入标灰”。

利用弱三色不变性消除了栈重扫

### GC触发的时机有哪些？

1. **主动触发**，通过调用 runtime.GC() 来触发 GC，此调用阻塞式地等待当前 GC 运行完毕。
2. **被动触发**，分为两种方式：
   - go后台有一系统监控线程，当**超过两分钟**没有产生任何 GC 时，强制触发 GC。
   - 内存使用**增长一定比例时**有可能会触发，每次内存分配时检查当前内存分配量是否已达到阈值（环境变量GOGC）：默认100%，即当内存扩大一倍时启用GC
     - 我们可以通过debug.SetGCPercent(500)来修改步调，这里表示，如果当前堆大小超过了上次标记的堆大小的500%，就会触发
     - 而第一次GC的触发的临界值是4MB

### error 和 panic

`error` 是一个**接口**。它是**业务逻辑的一部分**，代表“可能会发生且需要处理的情况”。
`panic` 是一个**内置函数**，它会**立即停止**当前协程的正常执行流程。它代表程序进入了**不可恢复的非法状态**。

panic后，只有 `defer` 里的代码会被执行，所以recover必须在 `defer` 中使用。且recover必须放在defer func() 中使用

### go的内存模型

​    Go 的内存模型规定了：**“在一个 Goroutine 中对变量进行写入，在什么条件下能保证被另一个 Goroutine 观察到。”**

#### Happens Before

​    Happens-Before 是一种“可见性保证”。如果事件 E_1 Happens-Before 事件 E_2，那么 E_2 的执行者一定能看到 E_1 产生的所有结果（比如对变量的赋值）。只有符合规则才能**确定代码执行顺序**

在 Go 中，以下是一些典型的 Happens-Before 规则：

1. 单协程规则：在同一个 Goroutine 中，代码逻辑上的顺序就是 Happens-Before 顺序。
2. Channel 规则（核心）：
   1. 发送 Happens-Before 接收完成。
   2. Channel 关闭 Happens-Before 接收到零值。
   3. 对于**无缓冲 Channel**，**接收完成** Happens-Before **发送完成**（强制同步）。
3. 锁规则：第 n 次 Unlock Happens-Before 第 n+1 次 Lock。
4. 启动规则：go 语句的执行 Happens-Before 该 Goroutine 内部的第一行代码。

如果能通过 Happens-Before 链条证明：写操作 $W \to$ 读操作 $R$，那么这个变量访问就是安全的。否则，你就必须加锁、使用 Channel 或 Atomic 操作。

根据这个模型，如果双重检查时不用原子操作，一句代码可能会被分开执行，也不是安全的。

### nil与零值的区别

1. 什么是零值 (Zero Value)？
   在 Go 中，当你**声明一个变量（所有类型）但不初始化**它时，Go 编译器会自动赋予该变量一个默认值，这就是 零值。但这个零值是 `nil` 还是具体的数值，完全取决于该变量的**数据类型**。结构体类型的零值是**一个其所有字段都为零值的实例**。
   
   每种基础类型都有自己的零值（非nil）：
   
   - 数值类型 (int, float, etc.): `0 或 0.0`
   - 布尔类型 (bool): `false`
   - 字符串类型 (string): `"" (空字符串)`
   - 复数类型: `0+0i`
   
   对于上述基础类型，它们永远不会是 nil。

2. 什么是 nil？
   nil 是 Go 中的一个预定义标识符，代表指针、接口、切片、映射、通道和函数的“零值”。
   
   它表示该变量目前**没有指向任何底层的内存地址或数据结构**。
   
   零值为 nil 的类型：
   
   - 指针 (*int, *Struct)，**`对nil指针进行读取/赋值成员、解引用会Panic`**
   - 切片 ([]int)，**`对nil 切片下标访问会Panic（越界）`**
   - 映射 (map[string]int)，**`对nil map写入会Panic，必须初始化。`**
   - 通道 (chan int)，**`关闭nil channel 会Panic`**
   - 接口 (interface{} / any)
   - 函数 (func())
   
   为什么对nil切片进行append不会Panic？ -  因为append检测到切片的指针是nil时，会自行去构造。
   
   为什么对nil map写入会Panic？ -  因为对map写入会根据其内的nil指针去寻找桶，所以Panic。

​    一个接口变量包含两个部分：Type（类型）和 Value（值）。 只有当 Type 和 Value **同时为 nil** 时，接口才等于 nil。

### 子协程panic为何没法被父协程recover

​    每个 Goroutine 都有自己独立的 调用栈（Stack）。recover 的机制是：**它只能捕获 当前 Goroutine 栈上的 panic**。

​    当子协程发生 panic 时，Go 运行时会开始“栈回溯”，试图在当前协程的栈中寻找 defer 函数里的 recover。如果回溯到栈顶还没找到，程序就会认为发生了不可恢复的错误，从而直接终止整个进程。

​    父协程此时正在自己的栈上运行，它完全感知不到另一个协程栈上的崩溃信号。

### Finalizer的作用

Finalizer（析构器）是一种特殊的机制，允许关联一个函数到一个对象上。当垃圾回收器（GC）发现该对象已经不可达，准备**回收其内存之前，会先调用这个关联的函数**。用于确保那些忘记手动关闭的资源最终能被释放。不建议使用。

```go
type singleton struct{}
```

双重检查锁定

```go
import (
    "sync"
    "sync/atomic"
)

type singleton struct{}

var (
    instance *singleton
    lock     sync.Mutex
    initialized uint32
)

func GetInstance() *singleton {
    // 第一次检查：如果已经初始化，直接返回
    if atomic.LoadUint32(&initialized) == 0 {
        lock.Lock()
        defer lock.Unlock()

        // 第二次检查：加锁后再判断一次，防止竞争，场景：协程a拿到lock，协程b等待lock，如果没有这个判断，协程a释放lock后，b又会再次创建实例
        if initialized == 0 {
            instance = &singleton{}
            atomic.StoreUint32(&initialized, 1)
        }

    }
    return instance
} 
```
